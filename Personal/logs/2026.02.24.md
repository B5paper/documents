* gRPC（推荐用于生产环境）

    gRPC 使用 Protocol Buffers (protobuf) 作为传输格式。它默认不支持 numpy，但你可以通过将数组转换为字节流（bytes）来传输。
    
    核心步骤：

    1. 定义 .proto 文件：

        ```Protocol Buffers
        syntax = "proto3";

        service DataService {
          rpc SendArray (ArrayRequest) returns (ArrayResponse);
        }

        message ArrayRequest {
          bytes data = 1;      // 序列化后的数组
          repeated int32 shape = 2; // 形状信息
          string dtype = 3;    // 数据类型
        }

        message ArrayResponse {
          string message = 1;
        }
        ```

    2. 序列化与反序列化：

        发送端： `data = arr.tobytes()`

        接收端： `arr = np.frombuffer(request.data, dtype=request.dtype).reshape(request.shape)`

* msgpack-numpy

* 压缩：如果网络带宽是瓶颈，可以在传输前使用 zlib 或 blosc 对 tobytes() 的结果进行压缩。

    ```py
    # 客户端发送
    import numpy as np
    import zlib

    def send_array(conn, arr):
        # 将数组转为字节流并压缩，避免 RPyC 的逐元素通信
        data = zlib.compress(arr.tobytes())
        shape = arr.shape
        dtype = str(arr.dtype)
        return conn.root.process_compressed_array(data, shape, dtype)
    ```

* 内存视图：在处理超大型数组时，尽量使用 memoryview 以减少内存拷贝。

* 极简 HTTP 接口	FastAPI + Msgpack	虽然不是传统 RPC，但在 Web 背景下传输 Numpy 非常方便。

    ```py
    arr.to_bytes()
    np.frombuffer()
    ```

    bytes 是内置类型，不需要任何特殊权限，不需要 allow_pickle。

* Netref (网络引用)

    当你通过 RPyC 传递一个 numpy 数组时，对方拿到的其实不是一个真正的数组，而是一个代理对象 (Proxy Object)。

    * 占位符：服务端只收到了一个指向客户端内存中该数组的“地址标签”。

    * 按需获取：

        * 如果你访问 arr.shape，RPyC 会发起一个小的网络请求问：“那个数组的 shape 是多少？”

        * 如果你访问 arr[0]，它会发起请求问：“那个数组的第 0 个元素是多少？”

        * 这种“懒加载”模式在处理超大对象且你只需要其中一小部分信息时非常高效。

    **为什么这对 Numpy 很危险？（性能杀手）**

    虽然“按需传输”听起来很智能，但在 numpy 这种密集计算场景下，它是性能噩梦。

    假设你有一个 100×100 的矩阵，你想在服务端计算总和：

    ```Python
    # 如果这是个 RPyC 代理对象
    total = 0
    for row in remote_arr:
        for val in row:
            total += val
    ```

    后果： 这段代码会产生 10,000 次网络请求。因为每一次 val 的提取都是一次 Transfer-on-Read。即使在局域网内（延迟 0.1ms），这段循环也需要 1 秒钟才能跑完，而本地计算只需要微秒级。

    **如何规避“按需传输”？**

    如果你确定需要处理整个数组，你应该强制进行一次性深拷贝 (Deep Copy)，将数据从代理对象转为本地对象：

    方法 A：使用 rpyc.classic.obtain()

    ```Python
    import rpyc
    local_arr = rpyc.classic.obtain(remote_arr) # 一次性把数据全拉回来
    ```

    方法 B：手动转换（最推荐）

    在发送前将其转为 bytes，接收后再 np.frombuffer。这样就跳过了 Netref 机制，直接走的是值的传输。

    总结一下： RPyC 默认确实是 Transfer-on-Read，它让远程调用看起来像本地调用一样“优雅”，但如果你在循环里读它，网络延迟会让你非常痛苦。

* 那我如果想机器a向b发送命令，修改的是b环境里的数据呢，类似ssh那种感觉

    如果你追求的是像 SSH 那样远程操控 B 机器的环境、修改 B 机器内存中的数据或调用其本地资源，那么 RPyC 的 "Classic" 模式 或 Paramiko (纯 SSH 实现) 是最匹配的方案。

    在这种场景下，你不再是“传输数据”，而是在“操作远程上下文”。

    1. 方案一：RPyC Classic (最像“远程 Python 解释器”)

        RPyC 有一种模式叫 classic，它允许你直接在 A 机器上操作 B 机器的 内建模块、变量 和 内存。

        服务端 (机器 B)

        只需启动一个通用代理服务：

        ```Bash
        rpyc_classic.py --allow-all-attrs  # 启动经典模式服务
        ```

        客户端 (机器 A)

        你可以在 A 上写代码，直接修改 B 的环境变量或内存对象：

        ```Python
        import rpyc

        # 连接到机器 B
        conn = rpyc.classic.connect("machine_b_ip")

        # 1. 像在 B 本地一样导入模块
        remote_os = conn.modules.os
        remote_np = conn.modules.numpy

        # 2. 修改 B 机器的环境变量
        remote_os.environ["MY_VAR"] = "changed_by_machine_a"

        # 3. 在 B 机器的内存里创建一个 Numpy 数组
        # 注意：这个数组存在 B 的内存里，A 只拿着一个引用
        remote_arr = remote_np.array([10, 20, 30])
        remote_arr[0] = 99  # 直接修改 B 内存中的数据

        print(f"B 机器上的数组第一个元素现在是: {remote_arr[0]}")
        ```

        特点：这种方式是真正的 "Remote Control"。你在 A 写的每一行指令，执行逻辑和数据存储都在 B 上。

    2. 方案二：Fabric / Paramiko (真正的 SSH 协议)

            如果你不仅想改 Python 里的数据，还想改 B 机器的文件、重启服务或执行 Shell 命令，应该用标准的 SSH 库。Fabric 是基于 SSH 的高级封装，非常适合自动化运维。

        示例 (机器 A 控制 B)

        ```Python
        from fabric import Connection

        # 建立 SSH 连接
        c = Connection("user@machine_b_ip")

        # 1. 执行 Shell 命令修改系统环境
        c.run("export DATA_PATH='/tmp/data' && echo $DATA_PATH")

        # 2. 甚至可以远程运行一段 Python 脚本来处理 B 上的 Numpy 数据
        remote_code = """
        import numpy as np
        data = np.load('data.npy')
        data = data * 2
        np.save('data_processed.npy', data)
        """
        c.run(f"python3 -c \"{remote_code}\"")
        ```

    3. 关键区别：你要修改的是什么？

        | 维度 | RPyC Classic | SSH (Fabric/Paramiko) |
        | - | - | - |
        | 操作对象 | 运行中的 Python 对象/内存 | 系统命令、文件、进程 |
        | 交互感 | 像在本地写 Python，实际在远程跑 | 像在写脚本，发送一串命令去执行 |
        | 性能 | 极高（保持长连接，对象操作快） | 较低（每次执行可能有握手开销） |
        | 持久性 | 服务端 Python 进程结束，数据消失 | 修改的是文件或系统状态，永久保留 |

    4. 安全性重要提醒（必读）

        由于你提到这像 SSH：

        * RPyC Classic 没有任何安全防护：它默认允许远程执行任何 Python 代码。千万不要将其暴露在公网，只能在可信局域网或 VPN 下使用。

        * 身份验证：SSH 方案（Fabric）原生支持密钥登录，安全性极高。

* RPyC 的“底层潜规则”

    1. exposed_ 前缀的真相（安全过滤）

        这是 RPyC 的一种命名空间保护机制。

        * 服务端规则：为了防止客户端通过 RPC 恶意调用服务端的内置方法（比如 __del__ 或 __init__），RPyC 规定：只有以 exposed_ 开头的函数，才允许被远程调用。

        * 客户端规则：客户端在调用时，必须省略这个前缀。当你调用 conn.root.process_array() 时，RPyC 的内核会自动在服务端寻找名为 exposed_process_array 的方法。

        如果你把服务端的 exposed_ 删掉，客户端就再也搜不到这个函数了。这相当于一种简单的 public/private 访问控制。

    2. root 到底是什么？

        root 是连接到远程服务的入口对象（也就是你定义的 MyService 实例）。

        在 RPyC 的协议里：

        * 当你执行 conn = rpyc.connect(...) 时，两端握手成功。

        * 服务端会实例化一个 MyService 类。

        * 客户端的 conn.root 实际上是一个代理对象（Proxy），它指向服务端那个 MyService 的实例。

        你可以把 conn.root 理解为**“那台机器上那个服务的根节点”**。所有你在服务端写的 exposed_xxx 函数，都会被挂载到这个 root 对象下面。

    3. 为什么不直接调，非要经过 root？

        这是为了支持多服务模型。

        虽然你现在只写了一个 MyService，但 RPyC 允许一个连接同时暴露多个不同的服务对象。root 默认指向主服务，确保路径清晰。

* 有哪些和 root 同级的概念？

    在 RPyC 的连接对象（conn）中，root 确实是最常用的入口，但从连接管理和协议控制的角度看，确实有几个和它同级（或者说在同一个层级由 conn 对象持有）的关键概念。

    理解了这些，你就能明白 RPyC 是怎么在底层“倒腾”数据的。

    1. conn.modules (最强大的同级属性)

        如果你开启了经典模式（Classic），conn.modules 是一个极其逆天的存在。它让你像操作本地模块一样操作远程模块。

        * 用法：conn.modules.os 或 conn.modules.numpy。

        * 对比：root 访问的是你自定义的业务逻辑（MyService），而 modules 访问的是远程机器整个 Python 环境。

        * 权限：它对权限要求极高，通常需要你在配置里显式允许。

    2. conn.builtins

        这也是一个代理对象，指向远程 Python 的内建命名空间（即 dir(__builtins__) 里的东西）。

        * 用法：如果你想在远程机器上调用 len()、open() 或 eval()，可以通过 conn.builtins.eval("1+1")。

        * 场景：当你想执行一些不属于任何模块的 Python 原生操作时。

    3. conn.root (深入理解)

        你已经知道了它是 Service 的实例。但要注意：root 是延迟加载的。
    当你第一次访问 conn.root 时，客户端才会请求服务端去创建那个 MyService 实例。
    4. conn._config (真正的配置所在地)

        这就是你刚才质问我的那个配置。

        在 conn 对象内部，确实有一个 _config 属性（注意带下划线）。这才是 RPyC 运行时真正叼的配置。

        * 真相：当你通过 rpyc.connect(..., config=my_config) 连接时，你的 my_config 会被合并进 conn._config。

        * 验证：你可以试着打印 print(conn._config)，你会看到一大堆默认值，包括你苦苦寻找的 allow_pickle、sync_request_timeout 等。

    5. conn.async_ (异步控制)

        如果你不想像现在这样等 process_array 执行完才返回（同步阻塞），你可以用 async_。

        * 用法：res = rpyc.async_(conn.root.process_array)(data)。

        * 状态：此时 res 会立即返回一个 AsyncResult 对象，你可以之后通过 res.ready 来判断算完没。

    总结对照表

    | 属性 | 作用范围 | 相当于 SSH 里的... |
    | - | - | - |
    | conn.root | 访问你写的 业务代码 (MyService) | 执行你写的远程脚本 |
    | conn.modules | 访问远程 第三方库 (numpy, os) | 远程库的交互式命令行 |
    | conn.builtins | 访问远程 原生函数 (print, eval) | 调用 Python 基础内核 |
    | conn._config | 控制这根 网线的属性 (超时、权限、序列化) | SSH 的 ssh_config 配置 |

* 简述 python 中的 try catch 如何使用

    在 Python 中，我们通常称之为 `try...except` 语句。它的核心逻辑是：**尝试执行一段可能出错的代码，如果真的出错了，就执行另一段代码来挽救，而不是让程序直接崩溃。**

    1. 基本语法结构

        Python 的异常处理非常灵活，最完整的结构包含四个部分：

        ```python
        try:
            # 可能会发生错误的代码
            result = 10 / 0
        except ZeroDivisionError:
            # 如果发生了指定的错误，执行这里的代码
            print("错误：除数不能为零！")
        else:
            # [可选] 如果 try 块中没有发生任何错误，执行这里
            print("计算成功，没有异常。")
        finally:
            # [可选] 无论是否发生错误，最终都会执行的代码（通常用于关闭文件或数据库）
            print("清理工作：操作已结束。")

        ```

    2. 核心用法详解

        **捕捉特定异常**

        不建议使用空的 `except:`，因为这会捕获包括“退出程序”指令在内的所有错误。明确指定错误类型（如 `ValueError`, `KeyError`）是更好的做法。

        **获取错误信息**

        你可以使用 `as` 关键字来获取具体的错误描述：

        ```python
        try:
            with open("config.txt") as f:
                data = f.read()
        except FileNotFoundError as e:
            print(f"找不到文件：{e}")

        ```

        **同时捕捉多个异常**

        如果不同的错误需要相同的处理方式，可以用元组包裹：

        ```python
        try:
            # 一些操作
            pass
        except (ValueError, TypeError):
            print("输入的数据类型有误。")

        ```

    3. 进阶：抛出异常 (`raise`)

        有时候你需要在代码中**主动**制造一个错误，告诉上层程序这里出问题了：

        ```python
        def set_age(age):
            if age < 0:
                raise ValueError("年龄不能为负数！")
            return age

        ```

    4. 最佳实践建议

        * **不要“吃掉”所有错误**：避免使用 `except Exception: pass`，这会让调试变得极其困难，因为你永远不知道哪里出了错。

        * **最小化 Try 范围**：只在可能出错的代码行周围使用 `try`，不要包裹整个长达百行的函数。

        * **利用 Finally**：它是确保资源（如网络连接或文件句柄）被释放的最可靠方法。

* 如果我在 except 时不知道具体的错误类型呢

    如果你在编写代码时，确实无法预知会发生哪种具体的错误，或者你希望通过一个“兜底”机制来确保程序不崩溃，可以使用 Python 的基类异常。

    这里有几种处理方式，按**推荐程度**排序：

    1. 推荐做法：使用 `except Exception as e`

        这是最标准的方法。`Exception` 是几乎所有常规错误的父类。通过 `as e`，你可以打印出具体的错误原因，而不至于两眼一抹黑。

        ```python
        try:
            # 可能出错的代码
            res = 1 / 0
        except Exception as e:
            # 打印错误类型和错误信息
            print(f"发生了一个预料之外的错误: {type(e).__name__} - {e}")

        ```

        * **优点**：能捕获绝大多数逻辑错误（如除零、下标越界、类型错误），同时还能保留错误信息。

        * **注意**：它不会捕获 `SystemExit` 或 `KeyboardInterrupt`（如用户按 Ctrl+C），这保证了你仍然可以强制停止程序。

    2. 进阶做法：查看堆栈轨迹 (`traceback`)

        当你不知道错误类型时，单纯打印错误信息有时不够，你可能还想知道错误发生在**哪一行**。

        ```python
        import traceback

        try:
            # 复杂逻辑
            do_something_risky()
        except Exception:
            # 打印完整的报错堆栈，就像程序崩溃时显示的那样
            traceback.print_exc()

        ```

        * **场景**：适用于开发调试阶段或记录详细日志。

    3. 不推荐的做法：空白 `except:`

        虽然在语法上合法，但非常不建议使用：

        ```python
        try:
            # 代码
        except:  # 极其危险！
            pass

        ```

        * **缺点**：它会捕获**所有**异常，包括你想要停止程序时的信号（Ctrl+C）。这可能导致你的程序陷入死循环，连点叉都关不掉，也就是俗称的“吃掉异常”。

    总结与建议

    | 方法 | 适用场景 | 评价 |
    | --- | --- | --- |
    | `except Exception as e` | 绝大多数情况 | **最平衡**，既安全又能获取信息 |
    | `traceback.print_exc()` | 调试、后台日志 | **最详细**，定位问题最快 |
    | `except:` | 极少数特定底层场景 | **最差**，容易掩盖真实的 Bug |

    **一个小技巧：**

    如果你在开发中发现自己经常使用 `except Exception`，建议先运行一次程序，观察控制台输出的错误类型（比如 `KeyError`），然后再回来把代码重构成具体的错误捕获，这样代码会更健壮。

* 简述 linux 中 screen 的用法

    简单来说，`screen` 是一个 **终端复用器**。它最核心的价值在于：即使你断开了 SSH 连接，原本在服务器上运行的任务也不会中断，你可以随时“重连”回去查看进度。

    安装：`sudo apt install screen`

    以下是 `screen` 的常用操作指南：

    1. 核心操作流程

        通常的使用逻辑分为三步：**创建 -> 挂起（离开）-> 恢复**。

        | 操作步骤 | 命令 / 快捷键 | 说明 |
        | --- | --- | --- |
        | **新建窗口** | `screen -S <名字>` | 创建并进入一个自定义命名的会话 |
        | **离会话** | `Ctrl + a`, 然后按 `d` | **保持任务运行**，并退回到主终端 (Detach) |
        | **查看列表** | `screen -ls` | 查看当前后台运行的所有会话及其 ID |
        | **恢复会话** | `screen -r <名字或ID>` | 重新进入之前挂起的会话 (Resume) |
        | **彻底关闭** | `exit` 或 `Ctrl + d` | 在会话内输入，任务结束并销毁该会话 |

    2. 常用快捷键（在会话内部使用）

        所有 `screen` 指令都以 `Ctrl + a` (通常简写为 `C-a`) 作为前缀。

        * `Ctrl + a` + `d`：脱离当前会话（最常用）。

        * `Ctrl + a` + `k`：强制杀掉当前窗口。

        * `Ctrl + a` + `[`：进入**复制模式**。你可以用方向键翻页查看历史滚屏记录，按 `Esc` 退出。

        * `Ctrl + a` + `c`：在当前会话中创建一个新的子窗口。

        * `Ctrl + a` + `n / p`：在多个子窗口间切换（Next / Previous）。

    3. 进阶技巧

        * **清理无效会话**：如果程序崩溃导致会话显示为 `Dead`，使用 `screen -wipe` 清除。

        * **强制接管**：如果会话显示 `Attached`（可能在另一台电脑连着），用 `screen -d -r <名字>` 强制将其在那头断开并在此处恢复。

        * **日志记录**：使用 `screen -L` 启动会话，可以将所有输出自动保存到日志文件中。

    常见场景示例

    如果你要运行一个耗时 10 小时的脚本：

    1. 输入 `screen -S train_model`。

    2. 在新窗口里执行 `./script.sh`。

    3. 按 `Ctrl + a`, `d` 潇洒下班，关掉笔记本。

    4. 第二天回公司，输入 `screen -r train_model` 查看结果。

* 那么如何在子窗口间切换？是否有类似 vim 那样的分栏显示的功能？

    在 `screen` 中，窗口切换和分栏（Split）是两个维度的操作。前者是在同一个屏幕里“切频道”，后者是将屏幕物理切分成多块。

    1. 窗口切换（切频道）

        如果你创建了多个子窗口（Window 0, 1, 2...），可以使用以下快捷键：

        * **快速轮换**：

            * `Ctrl + a` 然后按 `n` (Next)：切换到下一个窗口。

            * `Ctrl + a` 然后按 `p` (Previous)：切换到上一个窗口。

        * **按编号跳跃**：

            * `Ctrl + a` 然后按 **数字键 (0-9)**：直接跳到对应的窗口。

        * **回看切换**：

            * `Ctrl + a` 然后按 `Ctrl + a`：在当前和上一次查看的两个窗口之间快速来回切换。

    2. 分栏显示（类似 Vim 的 Split）

        `screen` 确实支持分栏，但逻辑和 Vim 略有不同：它先切分“区域（Region）”，然后你需要把“窗口（Window）”填进去。

        * **水平分栏**：`Ctrl + a` 然后按 `S` (大写 S)。

        * **垂直分栏**：`Ctrl + a` 然后按 `|` (竖线键，通常是 `Shift + \`)。

        * **移动光标**：`Ctrl + a` 然后按 `Tab`。光标会在不同的分栏区域间跳动。
        * **填入内容**：

            刚分好的栏是**空白**的。你需要跳到那个空白区域，然后：

            * 按 `Ctrl + a` 然后按 `c`：在那里创建一个新窗口。

            * 或者按 `Ctrl + a` 然后按数字（如 `0`）：把已有的窗口调到这个区域显示。



        **关闭分栏：**

        * **关闭当前栏**：`Ctrl + a` 然后按 `X` (大写 X)。

        * **只留当前栏（关闭其他所有）**：`Ctrl + a` 然后按 `Q` (大写 Q)。

    3. Screen vs Tmux：一个真诚的建议

        虽然你在问 `screen`，但不得不提的是，如果你对**分栏**、**鼠标拖拽**、**持久化布局**有较高要求，**Tmux** 是目前更现代、更强大的选择。

        | **垂直分栏** | 部分老版本不支持 | 原生完美支持 |
        | **分栏持久化** | 重连后布局常消失 | 配合插件可完美恢复布局 |
        | **状态栏** | 需复杂配置 | 默认自带，信息丰富 |
        | **学习曲线** | 略低 | 略高（但回报大） |

* 简述 bash 中 alias 的用法

    在 Bash 中，`alias`（别名）是一个非常实用的功能。简单来说，它允许你为冗长的命令设定一个**简短的“绰号”**，从而提高敲代码的效率。

    1. 基本语法

        设置别名的标准格式如下：

        ```bash
        alias 别名='实际执行的命令'

        ```

        > **注意**：等号 `=` 两边**不能有空格**，否则 Bash 会报错。

    2. 常用操作

        | 操作类型 | 命令示例 | 说明 |
        | --- | --- | --- |
        | **创建别名** | `alias ll='ls -alF'` | 输入 `ll` 即可执行详细列出文件的命令 |
        | **查看单个别名** | `alias ll` | 查看 `ll` 具体指向什么命令 |
        | **查看所有别名** | `alias` | 直接输入 `alias` 不带参数，列出当前所有已定义的别名 |
        | **删除别名** | `unalias ll` | 取消 `ll` 这个别名 |
        | **临时跳过别名** | `\ls` | 在命令前加反斜杠，强制执行原始程序而非别名 |

    3. 让别名永久生效

        如果你直接在终端输入 `alias` 命令，它只在**当前会话**有效。一旦关闭终端或重启，别名就会消失。

        要让它永久生效，需要将其写入 Bash 的配置文件中：

        1. 打开文件：`nano ~/.bashrc` (或使用 `vim`)。

        2. 在文件末尾添加你的别名，例如：

            ```bash
            alias gs='git status'
            alias ..='cd ..'
            alias py='python3'
            ```


        3. 保存退出后，执行以下命令立即使其生效：

            `source ~/.bashrc`

    4. 经典用法示例

        这里有一些大家公认的“效率利器”：

        * **安全保护**：防止误删重要文件。

            `alias rm='rm -i'` (删除前确认)

        * **简化路径**：快速回退目录。

            `alias ..='cd ..'`

            `alias ...='cd ../..'`

        * **网络检查**：

            `alias myip="curl ifconfig.me"`

        * **组合命令**：

        `alias update='sudo apt update && sudo apt upgrade'`

    ⚠️ 注意事项

    1. **参数处理**：`alias` 只能简单替换命令头。如果你需要处理复杂的参数（例如：把参数插在命令中间），建议使用 **Bash 函数 (Function)**。

    2. **优先级**：别名的优先级高于系统内建命令和可执行文件。如果你定义了 `alias ls='echo hi'`，那么原本的 `ls` 就被“遮住”了。

* python 2 不支持中文字符

    这个错误非常经典，它说明你正在使用 Python 2 运行含有**中文（或非 ASCII 字符）**的代码。

    **根本原因**

    Python 2 默认使用 ASCII 编码。当它在代码文件（比如第 11 行）遇到中文注释或中文字符串时，因为它不认识这些字节（\xe6 是 UTF-8 中文的起始字节），所以直接报错。

    Python 3 默认使用 UTF-8 编码，通常不会遇到这个问题。

    **解决方法**

    在文件开头声明编码（如果你必须用 Python 2）

    在 main.py 的第一行或第二行添加以下特殊注释，告诉解释器用 UTF-8 解析文件：

    ```Python
    # -*- coding: utf-8 -*-
    ```

* screen 如何上下滚动查看 terminal 输出

    在 `screen` 中，你不能直接用鼠标滚轮或平常的 `Shift + PageUp/PageDown` 来滚动屏幕，因为 `screen` 接管了终端的显示缓存。

    要查看历史输出，你需要进入 `screen` 的 **“复制模式”（Copy Mode）**。

    1. 核心操作步骤

        1. **进入模式**：按下 **`Ctrl + a`**，然后按 **`[`** (左方括号)。

            * 此时你会发现光标变成了一个可以移动的方块，屏幕右上角可能会出现类似 `Copy mode` 的提示。


        2. **上下滚动**：

            * **方向键**：上/下/左/右 逐行移动。
            * **翻页键**：`PageUp` 和 `PageDown`。
            * **类 Vim 快捷键**（推荐）：

                * `k` / `j`：上 / 下移一行。
                * `u` / `d`：向上 / 向下翻半屏 (Up / Down)。
                * `b` / `f`：向上 / 向下翻全屏 (Back / Forward)。
                * `g`：跳到文件最开头；`G`：跳到文件最末尾。

        3. **退出模式**：按 **`Esc`** 键（连按两次更保险）或 `Enter`。

            * 退出后，光标会回到最底部，恢复正常的命令行输入状态。

    2. 为什么我滚不上去？（缓存设置）

        如果进入模式后发现只能向上翻几行就到头了，说明 `screen` 默认的**回滚缓冲区（Scrollback Buffer）**太小了。

        **临时解决方法**：

        启动 screen 时指定缓冲区行数（例如 10000 行）：

        ```bash
        screen -h 10000
        ```

        **永久解决方法**：

        修改你的 `~/.screenrc` 文件（如果没有就创建一个）：

        ```bash
        nano ~/.screenrc
        ```

        在文件中添加这一行：

        ```bash
        defscrollback 10000
        ```

        保存退出。以后新建的每一个 screen 窗口都会拥有 10000 行的历史记录。

    3. 进阶技巧：搜索关键词

        进入 `Copy mode` (`Ctrl + a` + `[`) 后，你可以像在 `less` 或 `vim` 中一样搜索内容：

        * 按 **`/`**：向下搜索字符串。
        * 按 **`?`**：向上搜索字符串。
        * 按 **`n`**：跳转到下一个匹配项。

    如果你使用的是 **Tmux**，操作逻辑非常相似：`Ctrl + b` + `[` 进入模式，然后可以用方向键或 Vim 键位滚动。

* 那么进入 copy 模式后，我该如何使得键盘像vim 那样复制内容？

    既然你已经进入了 **Copy Mode** (`Ctrl + a` + `[`），`screen` 本身的设计逻辑就非常接近经典的 `vi`。

    以下是在 `screen` 中像 Vim 一样“选取、复制、粘贴”的标准动作：

    1. 像 Vim 一样复制 (Copy)

        在 Copy Mode 状态下（光标可自由移动时）：

        1. **定位起始点**：使用 `h/j/k/l` 或方向键将光标移动到你想复制的文字开头。
        2. **标记开始 (Set Mark)**：按下 **`Space`** (空格键)。

            * 你会发现光标下的字符被反色显示，这表示选区开始了。


        3. **移动选区**：继续移动光标，你会看到被选中的部分变色。

        4. **完成复制 (Yank)**：再次按下 **`Space`** (空格键) 或 **`Enter`**。
            * 选中的内容现在已经存入 `screen` 的内部剪贴板（Exchange Buffer）中。

            * 此时会自动退出 Copy Mode，回到正常终端状态。

    2. 像 Vim 一样粘贴 (Paste)

        要把刚才复制的内容贴出来：

        * 按下快捷键：**`Ctrl + a`** 然后按 **`]`** (右方括号)。

            * 这就相当于 Vim 里的 `p`。

    3. 高级技巧：Vim 党必备

        如果你习惯了 Vim 的快捷键，在 Copy Mode 里这些也是通用的：

        * **`0` / `$**`：跳到行首 / 行尾。
        * **`w` / `b**`：按单词向后 / 向前跳。
        * **`g` / `G**`：跳到缓冲区最顶端 / 最底端。
        * **`/`**：向下搜索；**`?`**：向上搜索。

    ⚠️ 一个常见的“坑”：剪贴板隔离

    **请注意**：`screen` 的这个复制粘贴是**它内部的**。

    * 这意味着你用 `Space` 复制的内容，**无法直接**在你的 Windows 或 macOS 桌面通过 `Ctrl + V` 粘贴出来。

    * 它只能在同一个 `screen` 会话的其他窗口（Window）里通过 `Ctrl + a` + `]` 粘贴。

* rpyc 版本对齐

    1. 为什么会报这个错？

        RPyC 协议在版本 4.x 到 5.x 之间发生过重大变化。

        18 这个消息类型代码在较新版本的 RPyC 中存在，但在旧版本（通常是 2.7 常用的 3.x 或 4.x 版本）的解析器里根本不存在。

        当你的 Python 3 客户端发送了一个“版本 5”特有的指令，Python 2 的服务端解析器直接就“懵了”，回了一个无效消息，导致客户端崩溃。

    2. 核心解决方案：对齐 RPyC 版本

        你不需要改变 Python 版本，但你必须强行统一两端的 RPyC 库版本。对于跨 2.7 和 3.x 的情况，4.1.2 通常是最稳定的中介版本。

    `pip install rpyc==4.1.2`

    3. 代码层面的关键兼容补丁

        由于 Python 3 默认传字符串是 unicode，而 Python 2 的 RPyC 4.x 可能会在反序列化时卡死，请在 Python 3 客户端的连接代码里加入一个特殊的“桥接配置”：

        ```Python
        import rpyc

        # 针对 4.1.2 版本的经典跨版本配置
        config = {
            "allow_all_attrs": True,
            "allow_pickle": True,
            "sync_request_timeout": 60,
            # 强制不进行自动编码转换，防止 Py3 的字符串传过去变成 Py2 无法识别的对象
            "instantiate_custom_exceptions": True,
            "instantiate_oldstyle_exceptions": True,
        }

        conn = rpyc.connect("localhost", 18861, config=config)
        ```

* 为什么会出现这个问题？

    pyenv 安装 Python 的本质是从源码现场编译。

    它就像一个厨师，如果发现你的厨房（系统）里没有 libsqlite3-dev（数据库原材料），它不会报错停止，而是选择“不做了”，直接跳过这个组件完成安装。

    这就是为什么你看到了 Installed 成功提示，但运行后却报 ImportError 的原因。
