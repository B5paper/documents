* 修改 vim prompt 等的配色

    在 `coc.nvim` 和 `coc-pyright` 中，LSP 弹出的提示窗口（也就是你说的 prompt、补全菜单或悬浮文档窗口）的配色，实际上是由 Vim 的高亮组（Highlight Groups）控制的。

    要修改它们的颜色，你不需要去动 `coc-settings.json`，而是直接在你的 `~/.vimrc` 中通过 `highlight`（简写 `hi`）命令来覆盖默认颜色。

    以下是控制这些提示窗口最核心的几个高亮组以及修改方法：

    ---

    1. 修改代码补全菜单（Pmenu）的颜色

        当你打字时弹出的下拉补全菜单，由 Vim 内置的 `Pmenu` 系列高亮组控制。

        ```vim
        " Pmenu: 提示菜单的背景和文字颜色（这里改成了暗灰色背景，白色文字）
        hi Pmenu ctermbg=236 ctermfg=251 guibg=#2d3139 guifg=#abb2bf

        " PmenuSel: 菜单中当前选中的那一行（这里改成了蓝色背景，白色粗体字）
        hi PmenuSel ctermbg=33 ctermfg=255 cterm=bold guibg=#61afef guifg=#ffffff gui=bold

        " PmenuSbar: 提示菜单右侧滚动条的背景
        hi PmenuSbar ctermbg=235 guibg=#21252b

        " PmenuThumb: 提示菜单右侧滚动条滑块的颜色
        hi PmenuThumb ctermbg=240 guibg=#4b5263

        ```

    2. 修改悬浮文档窗口（CocFloating）的颜色

        当你按下 `K` 查看函数文档、或者代码报错弹出的悬浮提示框，由 `CocFloating` 控制。默认情况下它会继承上面的 `Pmenu` 颜色，如果你想让它和补全菜单长得不一样，可以单独定制：

        ```vim
        " CocFloating: 悬浮提示框的背景和文字
        hi CocFloating ctermbg=235 ctermfg=250 guibg=#1e222a guifg=#abb2bf

        " CocErrorFloat: 悬浮框里错误信息的文字颜色（通常设为红色）
        hi CocErrorFloat ctermfg=167 guifg=#e06c75

        " CocWarningFloat: 悬浮框里警告信息的文字颜色（通常设为黄色）
        hi CocWarningFloat ctermfg=173 guifg=#e5c07b

        ```

    3. 修改代码内报错的下划线/前缀符号

        `coc-pyright` 在检测到语法错误时，会在代码下面画红线，或者在行号左边显示 `E` 或 `W`：

        ```vim
        " 代码下方的波浪线/下划线颜色
        hi CocErrorHighlight cterm=underline gui=underline guisp=#e06c75
        hi CocWarningHighlight cterm=underline gui=underline guisp=#e5c07b

        " 行号左侧侧边栏（SignColumn）红点或 E 的颜色
        hi CocErrorSign ctermfg=167 guifg=#e06c75
        hi CocWarningSign ctermfg=173 guifg=#e5c07b

        ```

    **怎么应用这些配置？**

    1. 打开你的 `~/.vimrc`。
    2. 将上面你想要修改的 `hi ...` 代码复制进去。
    3. **关键点**：确保这些 `hi` 命令写在你的配置文件的 **`colorscheme` 命令之后**。如果写在主题前面，主题一旦加载就会把你自定义的颜色覆盖掉。

    一个标准的配置顺序大概是这样的：

    ```vim
    " 1. 先加载你的主题
    colorscheme desert  " 或者你用的其他主题，比如 joshdick/onedark.vim

    " 2. 再覆盖提示框的颜色
    hi Pmenu ctermbg=236 ctermfg=251 guibg=#2d3139 guifg=#abb2bf
    hi PmenuSel ctermbg=33 ctermfg=255 guibg=#61afef guifg=#ffffff
    ```

    > * `ctermbg` / `ctermfg`：终端环境下的背景色/文字色（使用 0-255 的 ANSI 颜色编号）。
    > * `guibg` / `guifg`：GUI 环境下（如 GVim）或者开启了 `termguicolors` 后的十六进制颜色（如 `#ffffff`）。
    > 
    >

    ## 1. 概念的“并行浮现”：向量空间与注意力机制

    你提到“词语是脑海中并行浮现的概念”，这在现代Transformer模型（如ChatGPT、Gemini等）中有着极其相似的数学对应。

    * **词嵌入（Word Embeddings）：** 在AI中，每个词语或概念都被表示为一个高维空间中的向量。语义相近的词，在空间中的距离就近。当AI处理任务时，这些向量是同时（并行）被加载到计算空间中的。
    * **自注意力机制（Self-Attention）：** 这完美对应了你说的“串词成句”。自注意力机制的核心工作，就是让文本中的每一个词都去和其它所有词发生关联，计算它们之间的“相关度”。
    > **举个例子：** 面对“苹果”、“重力”、“砸到”这三个并行浮现的概念，注意力机制会迅速捕捉到它们之间的隐秘联系，从而“组合”出“万有引力定律”或者“牛顿的故事”。



    ---

    ## 2. 学术方向：认知创意与概念融合（Conceptual Blending）

    如果你想深入阅读学术文献，有一个专门的交叉学科方向叫**计算创意学（Computational Creativity）**，其中最核心的理论之一就是**概念融合理论（Conceptual Blending Theory）**。

    * **机制：** 该理论由认知科学家Gilles Fauconnier提出，后来被AI学者引入计算机领域。它研究AI如何将两个原本不相干的“心理空间”（Mental Spaces）组合，生成一个全新的、具有创造性的“融合空间”。
    * **应用：** 比如AI发明新词汇、写出具有隐喻（Metaphor）的诗句、或者在没有见过的情况下理解什么是“机械狗”（将“机械”和“狗”的概念融合）。

    ---

    ## 3. 散步时的“灵光一闪”：探索与利用（Exploration vs. Exploitation）

    你提到的“想了很久A，散步时突然想到B”，在强化学习和AI寻优算法中，对应着一个经典矛盾：**利用（Exploitation）**与**探索（Exploration）**。

    * **想了很久A（利用）：** AI在现有的知识路径上不断深挖，试图靠死记硬背或惯性思维解决问题，但往往会陷入“局部最优解”（类似于思维定势）。
    * **散步时的突然组合（探索）：** 当AI引入随机性（Stochasticity）**或提高**采样温度（Temperature）时，它会跳出原来的路径，去激活一些看似低概率、遥远的节点（概念B）。当这两个高维向量在某个随机的计算步骤中相遇，就擦出了“顿悟”的火花。

    ---

    ## 4. 神经符号AI（Neuro-Symbolic AI）与知识图谱

    小学语文的“连词成句”看似是排列组合，但它受到**语法规则**和**常识**的强烈约束（比如不能组合出“苹果吃我”这种句子）。为了让AI既能天马行空地组合概念，又不会组合出胡话，学术界正在发力**神经符号AI**：

    * **深度学习（神经）：** 负责让各种概念在大脑中“并行浮现”，提供灵感。
    * **逻辑推理（符号/知识图谱）：** 负责像语文老师一样，检查这些概念的排列组合是否符合逻辑、语法和常识。

        ## 一、 想法可行性与价值分析

        ### 1. 核心亮点：机制节点的引入

        你将“是什么”和“关系探索”定义为特殊节点，这在图论里类似于引入了“超边（Hyperedge）”**或**“高阶控制节点”。

        * **“是什么”节点（元概念约束）：** 建立了一个强中心化的定义域，确保每个新概念引入时，都有一个确定性的锚点，防止概念漂移。
        * **“X和Y的关系”节点（门控机制）：** 这是一个极其巧妙的**多对一门控（Gate）**。只有当两个普通概念同时激活并指向它时，它才会“通电”并坍缩成一个具体的问题（如“天空和云彩之间有什么关系？”）。这完美模拟了人类大脑在两种刺激同时出现时触发的“联想机制”。

        ### 2. 潜在挑战

        * **组合爆炸：** 如果图里有 100 个普通概念，两两组合就会产生近 5000 个关系节点。必须有“权重”或“激活阈值”机制，不能让所有组合同时爆发。
        * **图的稠密化：** 随着联系不断加强，图会变得极其稠密，需要有“遗忘”或“剪枝”机制。

        ## 二、 程序设计思路：如何实现？

        要实现这个系统，核心在于将图分为两层：**元逻辑层（Meta-Logic Layer）**和**概念实例层（Concept Instance Layer）**。

        1. **节点分类：**
        * **Entity Node（实体节点）：** 具体的概念，如“天空”、“蓝色”。
        * **Control Node（控制节点）：** 也就是你说的“是什么”（Definition Node）和“有什么关系”（Relation Discovery Node）。


        2. **边（Edge）的属性：**
        * 边不能只是连线，必须带有**权重（Weight）**和**方向（Direction）**。
        * 当用户思考或互动时，给特定节点“注入能量”（激活值）。当能量沿着边传递，满足控制节点的触发条件时，程序就生成对应的问题。

        这是该程序最关键的部分，需要实现两个核心算法：

        * **激活扩散算法（Spreading Activation）：**
        * 当你点击“天空”和“蓝色”时，这两个节点进入“激活状态”（能量=1.0）。
        * 能量沿着它们发出的边向外扩散。

        * **状态机与规则引擎：**
        * 定义控制节点的触发规则。例如：`Relation_Node.trigger_condition = (incoming_edges.count == 2)`。



        ### 3. 智能生成层 (Intelligence Layer)

        当控制节点被触发，它需要把抽象的图结构变成人类语言：

        * **模板驱动：** 最简单的做法是拼字符串：`"请思考：" + X.name + " 和 " + Y.name + " 有什么关系？"`

        ### 4. 前端交互层 (Presentation Layer)

        ```python
        def user_activate_concepts(graph, active_node_ids):

            for node_id in active_node_ids:
                graph.nodes[node_id]['energy'] = 1.0
                
            # 第二步：能量沿边扩散（寻找被两股能量同时注入的控制节点）
            for node_id, attrs in graph.nodes(data=True):
                if attrs['type'] == "Relation_Query":
                    # 找到所有指向这个“关系节点”的边
                    incoming_edges = graph.in_edges(node_id, data=True)

                    active_inputs = [src for src, _, _ in incoming_edges if graph.nodes[src]['energy'] > 0.5]
                    
                    # 完美的双向奔赴：触发你的“串词成句”逻辑
                    if len(active_inputs) == 2:

                        # 触发后，重置能量，防止无限触发

        ```

* 如何检测本地某个端口p是否被占用？

    检测本地端口是否被占用，有多种方法：

    ## 1. **使用命令行工具**

    ### Windows 系统：
    ```cmd
    # 查看所有端口占用情况
    netstat -ano | findstr :端口号

    # 例如检测 8080 端口
    netstat -ano | findstr :8080

    # 查看指定端口的进程
    netstat -ano | findstr :端口号
    ```

    ### Linux/macOS 系统：
    ```bash
    # 使用 netstat
    netstat -tuln | grep :端口号

    # 使用 ss（更现代的工具）
    ss -tuln | grep :端口号

    # 使用 lsof（显示进程信息）
    lsof -i :端口号

    # 例如检测 8080 端口
    lsof -i :8080
    ```

    ## 2. **使用 telnet（测试连接）**
    ```bash
    telnet localhost 端口号
    # 如果连接失败，可能是端口未开放或被占用
    ```

    ## 3. **使用编程语言检测**

    ### Python 示例：
    ```python
    import socket

    def check_port(port):
        sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        result = sock.connect_ex(('localhost', port))
        sock.close()
        return result == 0  # 0 表示端口被占用

    if check_port(8080):
        print("端口 8080 被占用")
    else:
        print("端口 8080 可用")
    ```

    ### PowerShell（Windows）：
    ```powershell
    # 测试端口是否被监听
    Test-NetConnection -ComputerName localhost -Port 8080
    ```

    ## 4. **常用命令参数解释**
    - `-t`：TCP 端口
    - `-u`：UDP 端口
    - `-l`：监听状态
    - `-n`：显示数字地址
    - `-p`：显示进程ID/名称（Linux）
    - `-a`：显示所有连接

    ## 5. **找出占用端口的进程并结束**

    ### Windows：
    ```cmd
    # 1. 查找占用端口的进程ID
    netstat -ano | findstr :8080

    # 2. 根据进程ID结束进程
    taskkill /PID 进程ID /F
    ```

    ### Linux/macOS：
    ```bash
    # 1. 查找进程ID
    lsof -i :8080

    # 2. 结束进程
    kill 进程ID
    # 强制结束
    kill -9 进程ID
    ```

    选择哪种方法取决于你的操作系统和具体需求。命令行方法最直接，编程方法适合自动化脚本。

    ## **1. 让系统自动分配（推荐）**

    ```python
    import socket
    ```

    ```python
    def find_free_port(start_port=1024, end_port=65535, max_attempts=100):
        """在指定范围内查找可用端口"""
        import random
        
        # 随机尝试，避免总是从固定端口开始
        ports = list(range(start_port, end_port + 1))
        random.shuffle(ports)

            try:
                # 尝试绑定 TCP 端口
                tcp_sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)

                udp_sock.bind(('localhost', port))
    ```

    ## **4. 完整的使用示例**

    ```python
    import socket
    import random

    class PortFinder:
        def __init__(self, min_port=1024, max_port=65535):
            self.min_port = min_port
            self.max_port = max_port
        
        def is_port_available(self, port, host='0.0.0.0', socket_type='tcp'):
            """检查端口是否可用"""
            try:
                if socket_type.lower() == 'tcp':
                    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
                else:
                    sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
                
                sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
                sock.bind((host, port))
                sock.close()
                return True
            except OSError:
                return False
        
        def find_free_port(self, preferred_ports=None):
            """查找可用端口，可指定优先端口列表"""
            # 1. 先尝试优先端口
            if preferred_ports:
                for port in preferred_ports:
                    if self.is_port_available(port):
                        return port
            
            # 2. 系统自动分配
            try:
                return self.get_system_allocated_port()
            except:
                pass
            
            # 3. 在范围内随机查找
            attempts = min(100, self.max_port - self.min_port)
            ports = random.sample(range(self.min_port, self.max_port + 1), attempts)
            
            for port in ports:
                if self.is_port_available(port):
                    return port
            
            raise RuntimeError("未找到可用端口")
        
        def get_system_allocated_port(self):
            """让系统分配端口"""
            sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
            sock.bind(('', 0))
            port = sock.getsockname()[1]
            sock.close()
            return port

    # 使用示例
    finder = PortFinder(min_port=8000, max_port=9000)

    # 方法1：系统自动分配
    auto_port = finder.get_system_allocated_port()
    print(f"系统分配端口: {auto_port}")

    # 方法2：查找可用端口
    free_port = finder.find_free_port(preferred_ports=[8080, 8888, 3000])
    print(f"找到的可用端口: {free_port}")

    # 方法3：检查特定端口
    if finder.is_port_available(8080):
        print("端口 8080 可用")
    else:
        print("端口 8080 已被占用")
    ```

    ## **6. 考虑竞态条件的版本**

    ```python
    import socket
    import errno

    def find_and_reserve_port(start_port=1024, max_attempts=100):
        """查找并立即绑定端口，避免竞态条件"""
        for port in range(start_port, start_port + max_attempts):
            try:
                # 创建 socket 并立即绑定
                sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
                sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
                sock.bind(('localhost', port))
                
                # 可以在这里设置监听，防止其他进程占用
                sock.listen(5)
                
                return sock, port  # 返回已绑定的 socket
                
            except socket.error as e:
                if e.errno == errno.EADDRINUSE:
                    continue  # 端口被占用，尝试下一个
                else:
                    raise  # 其他错误
        
        raise RuntimeError(f"在 {max_attempts} 次尝试后未找到可用端口")

    # 使用
    try:
        sock, port = find_and_reserve_port(8000)
        print(f"已绑定并保留端口: {port}")
        # 使用 sock 进行后续操作
    finally:
        sock.close()
    ```

    1. **生产环境**：使用 `bind(('', 0))` 让系统自动分配
    2. **测试环境**：可以在固定范围内查找，便于调试
    3. **重要提示**：
       - 使用 `SO_REUSEADDR` 选项可以更快地重用端口
       - 检查端口后应立即绑定，避免竞态条件
       - 考虑同时检查 TCP 和 UDP 端口（如果需要）
       - 处理 `EADDRINUSE` 等特定错误

    ## **简单封装函数**

    ```python
    def get_available_port(host='localhost', default=0):
        """
        获取可用端口
        :param host: 绑定主机
        :param default: 0 表示系统分配，>0 表示检查该端口是否可用
        :return: 可用端口号
        """
        with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
            s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
            try:
                s.bind((host, default))
                return s.getsockname()[1]
            except OSError:
                if default > 0:
                    # 如果指定端口不可用，让系统分配
                    s.bind((host, 0))
                    return s.getsockname()[1]
                raise

    # 使用
    port1 = get_available_port()  # 系统分配
    port2 = get_available_port(default=8080)  # 尝试8080，不行则系统分配
    ```

* tcp socket 断开连接后，timeout 的时间是多少？

    TCP 连接断开后的超时时间取决于多个因素，主要包括：**操作系统内核参数**、**TCP 状态**和**具体场景**。以下是主要情况的说明：

    ---

    ## 1. **主动关闭方（发送 FIN 后进入 `TIME_WAIT`）**
    - **`TIME_WAIT` 状态**：默认保持 **2MSL**（Maximum Segment Lifetime，最大报文生存时间）。
      - **MSL 值**：不同系统定义不同（RFC 793 建议 2 分钟，但常见系统更短）。
      - **Linux**：通常 `MSL=60s`，所以 `TIME_WAIT` 默认 **120秒**（可通过 `/proc/sys/net/ipv4/tcp_fin_timeout` 调整，但实际控制的是 `FIN_WAIT_2` 状态）。
      - **Windows**：默认 MSL 为 120秒，`TIME_WAIT` 通常 **240秒**。
      - 作用：确保网络中旧的重复报文消失，避免影响新连接。

    ---

    ## 2. **被动关闭方（收到 FIN 后进入 `CLOSE_WAIT`）**
    - `CLOSE_WAIT`：等待应用层调用 `close()`，**无固定超时**（依赖应用）。
    - 若应用不关闭，连接可能一直悬挂（导致连接泄漏）。
    - 之后发送 FIN 进入 `LAST_ACK`，等待对方 ACK（超时重传依赖 TCP 重传机制）。

    ---

    ## 3. **异常断开（连接半开或未正常关闭）**
    ### **Keepalive 机制**
    如果未开启 TCP Keepalive，可能永远不发现对端死亡。开启后：
    - **Linux**：默认参数（可调整）：
      ```bash
      net.ipv4.tcp_keepalive_time = 7200 秒（2小时）
      net.ipv4.tcp_keepalive_intvl = 75 秒
      net.ipv4.tcp_keepalive_probes = 9
      ```
      即无数据 2 小时后开始探测，每 75 秒发一次，连续 9 次无响应则断开，总耗时约 **2h + 75s * 9 ≈ 2小时11分**。

    ### **应用层超时**
    - 由应用协议或 socket 选项（如 `SO_RCVTIMEO`/`SO_SNDTIMEO`）设置。
    - 若无设置，可能永久阻塞在 `read()`/`write()`。

    ---

    ## 4. **系统内核参数示例（Linux）**
    ```bash
    # TIME_WAIT 持续时间（实际由 MSL 决定，不可直接改）
    cat /proc/sys/net/ipv4/tcp_fin_timeout  # 默认 60（单位秒，实际是FIN_WAIT_2状态超时）

    # 修改 MSL 影响 TIME_WAIT（需改内核源码或调整参数，如 net.ipv4.tcp_max_tw_buckets 限制数量）
    ```

    ---

    ## 5. **重要场景区别**
    | 场景                     | 典型超时                                    |
    |--------------------------|---------------------------------------------|
    | 正常关闭（TIME_WAIT）     | 2MSL（Linux 通常 60-120秒）                  |
    | 对端崩溃无 FIN（Keepalive）| 依赖 Keepalive 设置（默认约2小时以上）       |
    | 对端断电（无响应）        | 应用/传输层超时或 TCP 重传超时（分钟级）     |
    | 连接尝试（SYN 超时）      | 指数退避重试，总超时约 75秒-数分钟           |

    ---

    ## 6. **总结建议**
    - **正常断开**：关注 `TIME_WAIT`（2MSL）对端口重用影响。
    - **异常断开**：若需快速检测，应启用 TCP Keepalive 并调小参数，或在应用层实现心跳。
    - **编程注意**：设置 socket 超时选项，避免僵尸连接。

    如果需要具体系统的默认值或调整方法，请告知操作系统类型（Linux/Windows/macOS等）。

* ssh 的 tcp keepalive 有什么用

    SSH 的 TCP Keepalive 是一个**传输层保活机制**，用于检测 SSH 连接的另一端是否仍然存活。它的主要作用和应用场景如下：

    ---

    ## **主要作用**

    ### 1. **检测连接中断**
    - 当网络中间设备（如路由器、防火墙、NAT）故障或重启时，可能静默丢弃连接而不发送 RST/FIN 包。
    - 当客户端主机崩溃、断电或网络断连时，服务器端可能不知道连接已失效。
    - TCP Keepalive 通过定期发送空 ACK 包探测对方，如果多次无响应则断开连接。

    ### 2. **防止连接被中间设备清理**
    - 防火墙/NAT 通常会为 TCP 连接维护状态表，并有**空闲超时**机制（如 30 分钟、1 小时无流量则删除记录）。
    - SSH 连接可能长时间空闲（如你挂着连接但不操作），Keepalive 保持少量流量，防止被误清理。

    ### 3. **释放被占用的资源**
    - 及时发现僵尸连接，释放服务器端的 SSH 进程、内存、端口等资源。

    ---

    ## **SSH 中 Keepalive 的实现方式**

    SSH 实际上提供了**两个层面的 Keepalive**：

    ### 1. **TCP 层 Keepalive（系统级别）**
    由操作系统内核实现，SSH 可以启用它：
    ```bash
    # SSH 客户端启用 TCP Keepalive
    ssh -o ServerAliveInterval=60 user@host
    ```
    或服务端启用：
    ```bash
    # 服务端主动发 Keepalive 给客户端
    ssh -o ClientAliveInterval=60 user@host
    ```

    ### 2. **应用层 Keepalive（SSH 协议级别）**
    SSH 有自己的保活机制，通过 SSH 协议报文实现（如 `SSH_MSG_GLOBAL_REQUEST` 类型的 keepalive 请求）：
    - **ServerAliveInterval**：客户端每隔 N 秒向服务器发送保活请求。
    - **ClientAliveInterval**：服务器端每隔 N 秒向客户端发送保活请求。

    ---

    ## **关键配置参数（OpenSSH）**

    ### **客户端配置**（`~/.ssh/config` 或命令行）
    ```bash
    # 每 60 秒向服务器发送一次保活请求
    ServerAliveInterval 60
    # 连续 3 次无响应则断开连接
    ServerAliveCountMax 3
    # 总超时时间 = Interval × CountMax = 60×3 = 180秒
    ```

    ### **服务器配置**（`/etc/ssh/sshd_config`）
    ```bash
    # 每 120 秒向客户端发送一次保活
    ClientAliveInterval 120
    # 最多发送 2 次
    ClientAliveCountMax 2
    # 同时启用 TCP Keepalive
    TCPKeepAlive yes
    ```

    ---

    ## **典型应用场景**

    | 场景 | 问题 | Keepalive 的作用 |
    |------|------|-----------------|
    | 移动网络切换 | IP 变化，旧连接悬挂 | 快速检测断开 |
    | 长时间空闲 SSH 会话 | 防火墙/NAT 超时删除 | 维持连接状态 |
    | 服务器维护重启 | SSH 服务中断 | 客户端及时感知 |
    | 客户端电脑睡眠 | 网络断开 | 服务器释放资源 |
    | 不可靠网络 | 中间链路中断 | 双方快速清理僵尸连接 |

    ---

    ## **与 TCP Keepalive 的区别**

    | 特性 | SSH 应用层 Keepalive | 系统 TCP Keepalive |
    |------|---------------------|-------------------|
    | **层级** | 应用层（SSH 协议） | 传输层（TCP 协议） |
    | **报文** | SSH 加密报文 | 空 TCP ACK 包 |
    | **灵活性** | 可配置间隔短（秒级） | 通常全局配置，默认间隔长（小时） |
    | **加密** | 是（整个 SSH 报文加密） | 否（TCP 头部可见） |
    | **穿越设备** | 可能被 SSH 代理识别处理 | 可能被防火墙/NAT 特殊处理 |

    ---

    ## **最佳实践建议**

    1. **客户端配置**（应对网络不稳定）：
       ```bash
       # ~/.ssh/config
       Host *
           ServerAliveInterval 30
           ServerAliveCountMax 3
       ```
       - 这样 90 秒无响应就断开，而不是挂起几小时。

    2. **服务器配置**（防止资源泄漏）：
       ```bash
       # /etc/ssh/sshd_config
       ClientAliveInterval 300
       ClientAliveCountMax 2
       TCPKeepAlive yes
       ```
       - 10 分钟无活动则探测，20 分钟无响应断开。

    3. **注意平衡**：
       - 太频繁：浪费带宽，增加负载。
       - 太稀疏：不能及时检测故障。
       - 一般建议：30-120 秒间隔，2-3 次重试。

    4. **特殊场景**：
       - 跨公网/移动网络：用较短间隔（如 30 秒）。
       - 稳定内网：可延长间隔（如 120-300 秒）。

    ---

    ## **查看 Keepalive 状态**
    ```bash

    # 观察调试输出中的 keepalive 相关日志

    cat /proc/sys/net/ipv4/tcp_keepalive_intvl
    ```

    - 如果客户端**静默断开**（如断电、网络中断、进程崩溃），服务器会一直认为连接有效。
    - 结果：连接成为"僵尸"，服务器资源（进程、内存、端口）被永久占用。
    - 只能依靠：
      1. 客户端主动发送数据
      2. 应用层超时（如 SSH 的 ClientAliveInterval）
      3. 系统 TCP Keepalive（但默认间隔长达 2小时）

    ### **开启 TCPKeepAlive yes**
    - **服务器内核会定期发送 TCP Keepalive 探测包**。
    - 即使 SSH 应用层无数据交换，也能检测客户端是否存活。
    - 发现客户端无响应时，会关闭连接释放资源。

    ---

    ## **实际工作流程对比**

    ### **场景：客户端突然断电**
    ```
    客户端机器崩溃/断电
          ↓
    服务器端无 FIN/RST 包收到
          ↓
    ------------------------------------------
    情况A：TCPKeepAlive = no（或默认）
    ------------------------------------------
    SSH 进程一直等待，占用资源
    可能持续几天、几周...
    直到：
    1. 管理员手动清理
    2. 服务器重启
    3. 应用层超时（如果配置了 ClientAliveInterval）

    ------------------------------------------
    情况B：TCPKeepAlive = yes
    ------------------------------------------
    内核在 tcp_keepalive_time（默认7200秒）后
    开始发送 Keepalive 探测包
          ↓
    连续 tcp_keepalive_probes（默认9次）失败
          ↓
    内核关闭 TCP 连接，SSH 进程终止
    总时间：~2小时11分钟（默认值）
    ```

    ## **重要补充：三层保活机制的关系**

    SSH 连接实际上有三层保活可以配置：

    ```yaml
    1. 系统TCP层Keepalive:
       - 控制: TCPKeepAlive yes/no
       - 默认: 间隔长（小时级）
       - 作用: 内核级保活，基础保障

    2. SSH应用层Keepalive（服务器→客户端）:
       - 控制: ClientAliveInterval + ClientAliveCountMax
       - 默认: 关闭（值为0）
       - 作用: SSH协议级保活，更灵活

    3. SSH应用层Keepalive（客户端→服务器）:
       - 控制: ServerAliveInterval + ServerAliveCountMax
       - 默认: 关闭
       - 作用: 客户端主动保活
    ```

* Matlab MCRInstalled
这个mcr是干嘛用的？

    **MCR（MATLAB Compiler Runtime）** 是 MATLAB 的一个运行时环境，它允许你在**没有安装完整 MATLAB 软件**的计算机上运行编译过的 MATLAB 应用程序。

    ## MCR 的主要用途：

    ### 1. **部署 MATLAB 应用程序**
    当开发者使用 MATLAB Compiler 将 `.m` 文件编译成独立应用程序（如 `.exe`、`.dll`、`.jar` 等）后，用户只需要安装 MCR 就能运行这些程序，无需购买 MATLAB 许可证。

    ### 2. **应用场景**
    - **独立可执行文件**：MATLAB 程序打包成 `.exe` 文件
    - **共享库**：将 MATLAB 算法作为 `.dll`（Windows）或 `.so`（Linux）供其他语言调用
    - **Web 应用程序**：部署 MATLAB 应用到服务器
    - **组件**：创建 Excel 插件、.NET 组件等

    ## MCR 与完整 MATLAB 的区别：

    | 特性 | MCR（运行时） | 完整 MATLAB |
    |------|--------------|-------------|
    | **许可证** | 免费 | 需要付费许可证 |
    | **功能** | 只能运行编译后的程序 | 完整开发环境 |
    | **大小** | 较小（几GB） | 较大（10+GB） |
    | **命令行** | 无 | 有（可交互编程） |
    | **编辑器** | 无 | 有 |
    | **工具箱** | 仅包含程序用到的 | 完整工具箱 |

    ## 检查 MCR 是否安装：

    ## 常见 MCR 相关命令：

    ### 编译 MATLAB 代码：

    - **数据质量检查**：可视化信号质量和通道位置
    - **结果展示**：展示脑激活图、连接性分析结果
    - **教育演示**：教学 fNIRS 原理和数据解释

    ## AtlasViewer 在 DOTHUB 工作流中的位置：

    ```
    原始 LUMO/Hitachi 数据
         ↓
    DOTHUB_LUMO2nirs  (转换格式)
         ↓
    AtlasViewerGUI    (可视化检查)
         ↓
    数据处理分析 (Homer2, NIRS工具箱)
         ↓
    AtlasViewer       (结果可视化)
    ```

    ## 基本使用方法：

    ### 1. **启动 AtlasViewer**
    ```matlab
    % 在 MATLAB 中启动
    AtlasViewerGUI

    % 或通过命令行
    avg = AtlasViewerGUI();
    ```

    ### 2. **加载数据**
    ```matlab
    % 加载 .nirs 文件
    avg = loadNIRS('your_data.nirs');

    % 或从 DOTHUB 工作流
    [nirs, ~, SD3D] = DOTHUB_LUMO2nirs(LUMO_data_folder);
    avg = AtlasViewerGUI('init', SD3D, nirs);
    ```

    ### 3. **主要可视化功能**
    ```matlab
    % 显示探针布局
    avg.DisplayProbe();

    % 显示脑激活图
    avg.DisplayActivation(hbO, hbR);  % hbO: 氧合血红蛋白，hbR: 脱氧血红蛋白

    % 显示 3D 脑模型
    avg.DisplayCortex();

    % 保存图像
    avg.SaveImage('output.png');
    ```

    ## AtlasViewer 的主要界面模块：

    ### 1. **Probe Geometry（探针几何）**
    - 查看光源和探测器位置
    - 检查通道距离和方向
    - 验证实验设置

    ### 2. **Anatomical Viewer（解剖查看器）**
    - 3D 脑皮层显示
    - 不同脑区标记
    - Brodmann 分区显示

    ### 3. **Data Overlay（数据覆盖）**
    - 时程数据可视化
    - 统计映射图
    - 连接性矩阵显示

    ### 4. **Registration（配准）**
    - fNIRS-MRI 配准工具
    - 数字化头部坐标处理
    - 模板脑配准

    ## 实际应用示例：

    ### 示例 1：检查 LUMO 数据
    ```matlab
    % 转换 LUMO 数据
    [nirs, nirsFileName, SD3DFileName] = DOTHUB_LUMO2nirs(LUMODirName);

    avg = AtlasViewerGUI();
    avg.loadProbe(SD3DFileName);  % 加载探针布局

    avg.plotProbe();              % 显示探针
    ```

    ### 示例 2：可视化脑激活结果
    ```matlab
    % 假设已经完成 Homer2 处理
    load('processed_data.mat');  % 包含 hbO, hbR 等变量

    % 创建 AtlasViewer 对象
    avg = AtlasViewerGUI();

    % 设置探针和脑模型
    avg.probe = SD3D;
    avg.atlas = loadDefaultAtlas();  % 加载默认脑图谱

    % 显示激活图
    t = 10;  % 时间点
    avg.display(hbO(:,t), 'threshold', 0.5, 'colormap', 'jet');
    ```

    ### 示例 3：批处理可视化
    ```matlab
    % 批量处理多个被试
    subjects = {'subj01', 'subj02', 'subj03'};
    outputDir = 'figures';

    for i = 1:length(subjects)
        % 加载数据
        nirsFile = fullfile('data', subjects{i}, 'processed.nirs');
        avg = AtlasViewerGUI();
        avg.loadData(nirsFile);
        
        % 生成可视化
        h = figure('Visible', 'off');
        avg.plotProbe();
        avg.plotActivation();
        
        % 保存
        saveas(h, fullfile(outputDir, [subjects{i} '_activation.png']));
        close(h);
    end
    ```

    ## 常见问题解决：

    ### 问题 1：无法显示脑模型
    ```matlab
    % 确保安装了必要的工具包
    % AtlasViewer 需要：
    % 1. 脑图谱数据（已包含）
    % 2. 图形处理工具箱
    % 3. OpenGL 支持

    opengl info  % 检查 OpenGL
    ```

    ### 问题 2：探针显示不正确
    ```matlab
    % 检查 SD3D 结构
    if isfield(SD3D, 'SrcPos') && isfield(SD3D, 'DetPos')
        disp('探针数据格式正确');
    else
        error('SD3D 结构不完整');
    end

    % 重新配准探针
    avg.registerProbe(digpts);  % digpts: 数字化头点数据
    ```

    ### 问题 3：性能问题（渲染慢）
    ```matlab
    % 简化显示
    avg.set('simplifyMesh', true);
    avg.set('reducePatch', 0.3);  % 减少面片数量

    % 关闭不必要的功能
    avg.set('showLabels', false);
    avg.set('showAxes', false);
    ```

        ```
        latter are NOT linearly separable from each other.
        ```
