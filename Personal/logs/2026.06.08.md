    你这个需求可以拆成一句话来理解：

    > **同一个 QEMU 虚拟机里，一部分 PCIe 设备做“直通（VFIO passthrough，使用宿主 IOMMU）”，另一部分设备仍然走“虚拟 IOMMU（vIOMMU）/普通虚拟化路径”。**

    这是**可以做到的**，而且配置点非常明确，但有几个容易踩坑的地方。

    一、先澄清几个“概念层面的误区”

    1. IOMMU 有两层

        * **宿主机 IOMMU（host IOMMU）**

          * 给 VFIO passthrough 用
          * 控制 *物理设备 → 虚拟机* 的 DMA 隔离
        * **虚拟机 IOMMU（vIOMMU）**

          * 给 *虚拟设备 / SR-IOV / 非直通设备* 用
          * 让 guest OS 里也能看到 IOMMU（intel-iommu / amd-iommu）

        **两者是正交的，不是“二选一”**

    2. 走 iommu passthrough” ≠ “guest 不要 IOMMU

        你现在真正想要的是：

        | 设备                           | 方式                       |
        | ---------------------------- | ------------------------ |
        | 03:00.0–06:00.0 (ConnectX-7) | **VFIO 直通（host IOMMU）**  |
        | 11:00.0–14:00.0 (205d:1100)  | **普通虚拟 PCI 设备 + vIOMMU** |

        ✔ **完全合法**
        ❌ 不需要为此拆 VM
        ❌ 不需要 per-device 开关 host IOMMU

    二、宿主机侧：VFIO 直通设备配置

    1. 只把 03:00.0–06:00.0 绑定给 vfio-pci

        ```bash
        # 查看 IOMMU group（必须每个设备独占或可接受）
        for d in 0000:03:00.0 0000:04:00.0 0000:05:00.0 0000:06:00.0; do
          echo "$d -> $(basename $(readlink /sys/bus/pci/devices/$d/iommu_group))"
        done
        ```

    2. 绑定 vfio-pci（示例）

        ```bash
        echo 0000:03:00.0 > /sys/bus/pci/devices/0000:03:00.0/driver/unbind
        echo 15b3 1019 > /sys/bus/pci/drivers/vfio-pci/new_id
        ```

        （ConnectX-7 的 PCI ID 以实际为准）

    三、QEMU 命令行的**关键配置**

    1. **启用 vIOMMU（必须）**

        否则 11:00.0–14:00.0 在 guest 里就“没有 IOMMU”

        ```bash
        -device intel-iommu,intremap=on,caching-mode=on
        ```

        （AMD 平台用 `amd-iommu`）

    2. **VFIO 直通设备：关闭 iommu_platform**

        ```bash
        -device vfio-pci,host=03:00.0,iommu_platform=off
        -device vfio-pci,host=04:00.0,iommu_platform=off
        -device vfio-pci,host=05:00.0,iommu_platform=off
        -device vfio-pci,host=06:00.0,iommu_platform=off
        ```

        解释：

        * `iommu_platform=off`

          * **设备 DMA 直接走 host IOMMU**
          * 不走 guest vIOMMU
          * 这是 *bare-metal 风格直通*（你要的）

    3. 普通 / 虚拟设备：默认走 vIOMMU

        例如：

        ```bash
        -device virtio-net-pci,netdev=net0
        -device virtio-blk-pci,drive=vd0
        ```

        或者你那个：

        ```text
        11:00.0 ~ 14:00.0  Processing accelerators
        ```

        如果它们是：

        * virtio
        * emulated PCI
        * mediated device (mdev)

        👉 **什么都不用加**
        👉 它们会自动挂到 `intel-iommu`

    四、Guest OS 内部的表现（你应该看到什么）

    1. IOMMU 存在

        ```bash
        dmesg | grep -i iommu
        ```

        ```text
        DMAR: IOMMU enabled
        ```

    2. VFIO 设备 **不受 guest IOMMU 约束**

        ```bash
        lspci -vvv -s 03:00.0
        ```

        * 不会看到 guest IOMMU fault
        * DMA 地址是 guest 物理地址 → host IOVA

    3. 11:00.0–14:00.0 受 guest IOMMU 约束

        ```bash
        ls /sys/kernel/iommu_groups/
        ```

        能看到这些设备被分组

    五、几个“非常重要”的坑（结合你之前 NCCL / RDMA 经验）

    1. **ConnectX-7 + iommu_platform=on 很容易炸**

        * MLX5 在 vIOMMU 下：

          * ATS / PRI
          * page fault
          * peer-direct
            都很容易触雷

        👉 **你选 `iommu_platform=off` 是对的**

    2. vIOMMU 不等于性能损失

        * virtio / accelerator 设备：

          * 本来就做 bounce / map
        * vIOMMU 只提供地址翻译
        * 不会影响 VFIO 设备

    3. 一个 VM 里“混用”是官方支持的

        这是 QEMU 官方文档里的推荐组合：

        > vfio-pci (iommu_platform=off)
        >
        > * intel-iommu
        > * virtio / emulated PCI

      -device vfio-pci,host=06:00.0,iommu_platform=off \

* qemu-system-x86_64: -device vfio-pci,host=3c:00.0,bus=ds1,iommu_platform=off: Property 'vfio-pci.iommu_platform' not found

    这个报错**非常关键**，而且信息本身已经把问题点出来了：

    > `Property 'vfio-pci.iommu_platform' not found`

    结论先给出来：

    > **你的 QEMU 版本里，`vfio-pci` 这个 device 根本还不支持 `iommu_platform` 这个属性。**
    > 所以这不是你命令写错，而是 **QEMU 太老 / 编译配置不包含该特性**。

    下面一步一步把来龙去脉说清楚，以及你现在**能怎么做**。

    一、`iommu_platform` 是什么时候引入的？

    1. 属性来源

        `iommu_platform` 是 **VFIO 的“DMA API 协议切换”开关**，用于区分：

        | 模式                   | 行为                                      |
        | -------------------- | --------------------------------------- |
        | `iommu_platform=off` | 设备 DMA **只用 host IOMMU**（bare metal 风格） |
        | `iommu_platform=on`  | 设备 DMA 走 **guest vIOMMU（DMA API）**      |

    2. 引入时间（关键）

        这个属性是在 **QEMU 6.x 之后** 才稳定出现的（5.x 很多发行版压根没有，或被 backport 裁掉）。

        如果你看到这个错误，基本可以断定：

        * QEMU ≤ 5.x
          **或者**
        * 发行版定制版（RHEL / CentOS / Rocky / Ubuntu LTS）
          👉 **编译时禁用了该属性**

    二、为什么你现在“其实不需要”这个参数？

    这是一个很多人会误判的点。

    **事实是：**

    > **在老 QEMU 里，VFIO 默认行为 = `iommu_platform=off`**

    也就是说：

    ```text
    -device vfio-pci,host=03:00.0
    ```

    在你现在的 QEMU 上，**等价于**：

    ```text
    -device vfio-pci,host=03:00.0,iommu_platform=off
    ```

    👉 **不会经过 guest vIOMMU**
    👉 DMA 直接走 host IOMMU
    👉 正是你想要的 ConnectX-7 行为

    三、那 guest 里的 vIOMMU 会不会“影响直通设备”？

    **不会。**

    这是另一个容易混淆的点。

    **行为矩阵（老 QEMU）**

    | 设备类型              | guest 有 intel-iommu | 实际 DMA                |
    | ----------------- | ------------------- | --------------------- |
    | vfio-pci          | 有                   | **不受 guest IOMMU 影响** |
    | virtio / emulated | 有                   | 走 guest IOMMU         |
    | vfio-pci          | 没有                  | 仍然走 host IOMMU        |

    👉 **guest IOMMU 只是给“虚拟设备”用的**

    四、所以你现在该怎么改？

    **正确做法（删掉参数）**

    ```bash
    -device vfio-pci,host=3c:00.0,bus=ds1
    ```

    不要写 `iommu_platform`。

    五、如何“确认”它确实没走 guest IOMMU？

    你可以用这几个方法自证，而不是靠猜。

    1. Guest 里打开 IOMMU 调试

        ```bash
        intel_iommu=on,debug
        ```

        如果 VFIO 设备走了 vIOMMU，你会看到：

        * DMA map/unmap
        * IOTLB flush
        * fault log

        👉 **你不会看到 ConnectX-7 的任何记录**

    2. mlx5 驱动行为验证（你这个背景很重要）

        结合你之前 **NCCL / RDMA / 非均匀 chunk** 的经历：

        * 如果走 vIOMMU：

          * 会触发 ATS / PRI
          * 性能下降明显
          * 甚至初始化卡死
        * 如果走 host IOMMU：

          * 行为和 bare metal 一致
          * `mlx5_ib` 初始化干净

        这在 ConnectX-6/7 上非常明显。

    六、什么时候你**必须**升级 QEMU？

    只有在你需要下面这种**混合精确控制**时：

    | 场景        | 需求                   |
    | --------- | -------------------- |
    | 同一个 VM    | **部分 VFIO 走 vIOMMU** |
    | SR-IOV VF | 需要 DMA API           |
    | 设备支持 PRI  | 想让 guest 管 DMA       |

    👉 才需要 `iommu_platform=on/off` 显式控制
    👉 那就得 **QEMU ≥ 6.2**（推荐 ≥ 7.x）
