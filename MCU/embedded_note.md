# Embedded Note

## cache

* 8259A 如果有多个中断到达，那么低优先级的中断会被缓存吗，比如等高优先级的中断处理完了，再处理低优先级的？

    这是一个非常好的问题，它触及了8259A可编程中断控制器（PIC）的核心工作原理。

    简短的回答是：**不会缓存，但会“记住”（Pending）**。

    8259A内部有一个名为**中断请求寄存器**（IRR, Interrupt Request Register）和一个**中断服务寄存器**（ISR, In-Service Register）。它的处理机制比简单的“缓存”要更精巧一些。下面我来详细解释这个过程：

    ### 详细工作流程：

    1.  **中断到达（Interrupt Request）**：
        *   当一个中断信号（例如来自IR0-IR7中的某一个）到达时，无论其优先级如何，8259A会首先在**IRR**中对应的位上进行标记（置1）。你可以把IRR看作一个“等待处理”的列表。

    2.  **优先级裁决（Priority Arbitration）**：
        *   8259A会持续检查IRR。如果发现有中断被标记，它会将其中**优先级最高**的中断挑选出来。
        *   然后，8259A会检查当前CPU是否正在处理一个中断（通过检查ISR寄存器）。如果没有，或者如果新中断的优先级**高于**当前正在处理的中断的优先级，8259A才会向CPU发送INT信号。

    3.  **CPU响应（CPU Acknowledges）**：
        *   CPU收到INT信号后，如果中断是允许的（IF标志位为1），它会通过INTA引脚发出两个中断应答周期。
        *   在第一个INTA周期，8259A会做两件事：
            a.  将ISR中对应此中断的位**置1**（标记为“正在服务”）。
            b.  将IRR中对应的位**清0**（从“等待”列表中移除）。
        *   在第二个INTA周期，8259A将对应此中断的**中断向量号**发送到数据总线上，CPU据此跳转到正确的中断服务程序（ISR）。

    4.  **处理多个中断（关键部分）**：
        *   在**非自动结束（Non-Automatic End of Interrupt, Non-AEOI）** 模式下（这是最常见和推荐的模式），ISR中被置1的位会**一直保持为1**，直到中断服务程序在结束时向8259A发送一个**EOI（End of Interrupt）命令**。
        *   在CPU处理高优先级中断的期间（ISR位为1，且未收到EOI）：
            *   **低优先级的中断**：它们会一直**保留在IRR中**（状态为Pending）。8259A知道它们来了，但因为当前ISR中有一个更高优先级的位正在被服务，所以8259A**不会**向CPU发送新的INT信号。它们会耐心地“等待”。
            *   **更高优先级的中断**：如果一个**更高优先级**的中断到达，它会被记录到IRR中。由于它的优先级比当前正在服务（ISR中）的中断高，8259A会**立即**再次向CPU发送INT信号，从而实现**中断嵌套**。

    5.  **结束中断（End of Interrupt）**：
        *   当高优先级的中断处理程序执行完毕，在返回前它会向8259A发送一个**EOI命令**。这个命令会**清除ISR中对应的位**（即告知8259A这个中断的服务已完成）。
        *   ISR中被清除后，8259A会再次检查IRR寄存器。此时，之前那个在“等待”的低优先级中断就成了IRR中优先级最高的请求。于是，8259A会**立刻**再次向CPU发送INT信号，请求处理这个低优先级的中断。

    ### 总结与类比：

    可以把8259A想象成一个公司的前台，而CPU是老板。

    *   **IRR**：是前台手上的**待办事项清单**。
    *   **ISR**：是前台墙上的**正在处理中**的清单。
    *   **优先级**：每个事项都有不同的紧急程度。

    1.  销售部（高优先级）和行政部（低优先级）同时提交问题（中断到达），前台都记在了待办清单（IRR）上。
    2.  前台看到老板（CPU）现在有空，于是先把最紧急的销售部问题（高优先级中断）报给了老板，并把该问题挪到“正在处理”的墙上（ISR），同时从待办清单上划掉（IRR清零）。
    3.  老板正在处理销售部的问题时，前台**不会**用行政部的问题去打断他（低优先级中断在IRR中Pending）。
    4.  老板处理完销售部的问题后，告诉前台：“这件事我办完了”（发送EOI），前台就把墙上“销售部问题”的纸条撕掉（ISR清零）。
    5.  前台一看待办清单（IRR），发现还有一个行政部的问题（低优先级中断），于是立刻再次报告给老板。

    **所以，结论是：低优先级的中断不会被“缓存”到某个队列后再按顺序处理，而是其请求状态（Pending）会一直被保留在IRR寄存器中，直到更高优先级的中断处理完毕、ISR被清空后，它才有机会被CPU响应和处理。** 这种机制确保了中断严格按照优先级顺序被处理，并且允许更高优先级的中断实现嵌套。

    * 也就是说，假如一个中断的IRR为1，此时又来了个相同的中断，那么第二个中断就被忽略掉了

        **完全正确！您的理解非常到位。**

        这正是8259A的一个重要行为特性。当一个中断请求已经被记录在IRR（Interrupt Request Register）中（即对应位为1），但还没有被CPU响应和处理时，如果**完全相同的中断源**再次发出请求，**第二个以及后续的请求将会被忽略**，直到第一个请求被处理完毕。

        ### 详细解释这个过程：

        1.  **第一个中断到达**：假设IRQ0（定时器中断）产生了一个上升沿或高电平（取决于触发方式）。8259A会在其IRR寄存器的第0位**置1**，表示“有一个IRQ0请求正在等待处理”。

        2.  **裁决与响应**：8259A进行优先级裁决。如果它是当前最高优先级且CPU允许中断，8259A会向CPU发送INT信号。CPU响应后，在第一个INTA周期，8259A会：
            *   将ISR（In-Service Register）的第0位**置1**（标记IRQ0正在服务）。
            *   将IRR的第0位**清0**（表示等待队列中的这个请求已被领取）。

        3.  **第二个相同中断到达**：**就在CPU正在执行IRQ0的中断服务程序（ISR）时**，同一个IRQ0线路上又来了一个新的中断脉冲。
            *   8259A检查IRR的第0位，发现它是**0**（因为在上一步已经被清除了）。
            *   8259A会**再次将IRR的第0位置1**，记录下这个新的请求。

        4.  **问题显现**：现在，ISR的第0位是1（表示IRQ0正在服务），IRR的第0位也是1（表示有一个新的IRQ0在等待）。
            *   然而，8259A的**优先级裁决器**会发现：当前正在服务的中断（ISR中为1的位）和**新来的等待中断是同一个优先级**（实际上是同一个引脚）。
            *   8259A的设计是：**它不会让同一个优先级的中断去打断它自己**。除非是更高优先级的中断，否则在收到EOI命令、清除ISR位之前，8259A**不会**再次为IRQ0向CPU发送INT信号。

        5.  **结果**：这个**第二个中断请求虽然被记录在了IRR中，但它永远无法被CPU响应**，因为CPU正在执行的就是它的服务程序。只有当CPU处理完当前的ISR并发送EOI命令后：
            *   ISR位被清除。
            *   8259A重新检查IRR，发现第0位是1，于是**立刻再次**向CPU发送INT信号，请求**第二次**处理IRQ0。

        ### 这意味着什么？（重要影响）

        *   **中断丢失（Interrupt Lost）**：如果中断产生的速度**快于**中断服务程序执行的速度，那么中间的中断就会被丢失。CPU只会处理一次，但实际上事件发生了多次。
        *   **对设计的要求**：这要求程序员必须意识到中断服务程序（ISR）**必须尽可能的短小精悍**，只处理最紧急的任务（如读取数据、清除中断源），否则很容易丢失后续的中断。对于高速设备（如网卡、硬盘），这通常是不可接受的。

        ### 如何解决这个问题？

        这正是现代系统中**更多使用本地APIC（Advanced Programmable Interrupt Controller）** 而不是8259A的原因之一。APIC支持一种叫做**电平触发（Level-Triggered）** 的模式。


        *   **在电平触发模式下**：中断信号不是靠一个脉冲，而是靠维持一条线的电平（通常是高电平）来表示请求。
        *   只要该线路的电平保持为高，就表示中断请求**持续有效**。CPU在中断处理结束后，会检查该线路。如果电平仍然是高的，说明又有新的请求，它会**再次触发中断**。这样就有效地避免了相同中断的丢失问题。

        **总结：您的说法完全正确。在边沿触发模式下，8259A无法缓存多个相同的未处理中断，后续的请求会被“忽略”或“合并”，直到当前中断处理完毕，这可能导致中断丢失。**

* 简述什么是高低速接口驱动开发工程师

    **高低速接口驱动开发工程师**是嵌入式系统/芯片开发领域的专业职位，主要负责在操作系统内核层（如Linux、RTOS、Android等）开发、移植和优化各种通信接口的驱动程序。其核心工作是确保硬件接口能够被上层软件正确、高效、可靠地访问和控制。

    **核心职责与工作内容**

    1. **低速接口驱动开发**

        *   **典型接口**：I²C、SPI、UART、GPIO、I²S（音频）、PWM等。

        *   **特点**：数据速率相对较低，通常用于控制、配置、状态读取或小批量数据传输。

        *   **工作重点**：保证稳定性、实时性、低功耗和精确的时序控制。常用于连接传感器、触摸屏、EEPROM、音频编解码器等外设。

    2. **高速接口驱动开发**

        *   **典型接口**：USB（2.0/3.x/4）、PCIe、以太网（MAC/PHY）、MIPI（CSI/DSI）、SD/eMMC、SATA、HDMI/DP等。

        *   **特点**：数据速率高（可达Gbps级别），协议复杂，对吞吐量、延迟、带宽利用率要求极高。

        *   **工作重点**：优化大数据流的DMA传输、内存管理、中断处理、电源管理和协议栈的完整实现。常用于连接摄像头、显示屏、存储设备、高速外设和网络。

    3. **公共及跨领域任务**

        *   **驱动框架适配**：遵循Linux内核等现有的驱动模型（如Platform Driver、PCI Driver、USB Gadget等）进行开发。

        *   **芯片移植与BSP开发**：为新芯片或开发板适配和实现所有必要的接口驱动，构成板级支持包。

        *   **调试与调优**：使用逻辑分析仪、示波器等工具进行硬件级调试；进行性能剖析与优化。

        *   **问题排查**：解决驱动与硬件、其他驱动或应用层之间的兼容性、稳定性问题。

        *   **文档编写**：撰写设计文档、API说明和调试指南。

    **必备技能栈**

    *   **核心语言**：精通 **C 语言**，熟悉汇编语言（特定架构）。

    *   **操作系统**：深入理解 **Linux 内核机制**（进程调度、内存管理、中断、DMA、并发与同步等）。

    *   **硬件知识**：能阅读 **硬件原理图** 和 **芯片数据手册**，理解接口的时序图、寄存器定义。

    *   **协议理解**：深入掌握所负责接口的 **通信协议标准**。

    *   **调试能力**：熟练使用内核调试工具（如 `printk`、`ftrace`、`kgdb`）、逻辑分析仪等。

    *   **版本管理**：熟练使用 Git。

    **在产业中的位置与价值**

    *   **位置**：处于**硬件（HW）** 与**上层软件/应用（SW）** 之间的关键桥梁，是系统能正常工作的基石。

    *   **价值**：

        *   **使能硬件**：没有稳定的驱动，再强大的硬件也无法被使用。

        *   **性能决定者**：驱动的质量直接决定了接口的实际性能上限和系统功耗。

        *   **系统稳定性保障**：驱动层的Bug常导致系统崩溃、死机，是系统稳定的关键。

    **简而言之**

    **高低速接口驱动开发工程师，就是让芯片上的各种通信“道路”（从乡间小路到信息高速公路）在操作系统中畅通无阻的“筑路工”和“交通规则制定者”。** 他们需要既懂硬件细节，又精通软件内核，是嵌入式领域技术要求高、不可或缺的核心角色。
