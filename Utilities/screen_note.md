## cache

* 讲讲 screen 中 defflow off 的作用

	`defflow off` 命令用于**关闭 GNU Screen 对`XON`/`XOFF`（即 `Ctrl-S` 和 `Ctrl-Q`）流量控制信号的拦截和处理**。简而言之，它告诉 Screen 不要"吃掉"这两个按键，而是将它们直接传递给在窗口中运行的程序。

	这个设置主要带来两方面的影响：

	### 核心作用：解决 `Ctrl-S` 的"假死"问题

	默认情况下，Screen 的流量控制是开启的（`defflow on` 或 `auto`）。当你在终端里按下 `Ctrl-S` 时，Screen 会拦截它，并立刻暂停（阻塞）当前窗口的所有输出，直到你按下 `Ctrl-Q` 来恢复。

	*   这个功能来自硬件终端时代，但现代环境下，`Ctrl-S` 常被误触，导致终端莫名其妙"卡住"（输出暂停）。
	*   设置 `defflow off` 后，Screen 不再处理这两个字符，`Ctrl-S` 按键会直接发送给当前程序，从而避免了这个烦人的问题。

	### 潜在代价：`Ctrl-S` 与文本编辑器的兼容

	关闭流量控制的代价是，你失去了一种快速暂停屏幕输出的方法。但对大多数用户来说，这不成问题。

	*   对于一些常用的文本编辑器（特别是 Emacs），`Ctrl-S` 是一个非常重要的快捷键（用于搜索）。
	*   如果在 Screen 中开启了流量控制，你在 Emacs 里按 `Ctrl-S`，Screen 会拦截它，导致搜索功能无法使用。设置 `defflow off` 可以让 `Ctrl-S` 正确传递给 Emacs。

	### 其他补充说明

	*   **设置方法**：将 `defflow off` 这一行添加到你的 `~/.screenrc` 配置文件中。
	*   **作用范围**：`defflow` 设置的是**新创建窗口**的默认流量控制状态，对已存在的窗口没有影响。
	*   **其他方案**：如果你依然希望保留流量控制功能，同时又想使用 `Ctrl-S`，可以修改 Screen 的默认转义键（例如改为 `Ctrl-J`），但这比直接关闭流量控制要复杂一些。

* 讲讲 screen 中 flow off 的作用

    在 GNU Screen 中，**`flow off`**（注意没有 `def` 前缀）与上一条回答中的 **`defflow off`** 有着本质区别。它们的作用对象和作用时机完全不同。

    简单来说：
    **`defflow off`** 是**全局默认设置**（影响未来新建的窗口）。
    **`flow off`** 是**即时操作命令**（影响当前正在使用的这个窗口）。

    ### 5. 如何查看当前窗口的状态

    如果你想确认当前窗口的流量控制是开还是关，可以在 Screen 命令行中输入：

    ```bash
    Ctrl-A : flow
    ```
    （不加 `on` 或 `off` 参数），Screen 会返回类似 `Flow is on` 或 `Flow is off` 的提示信息。

* 讲讲 screen 中 termcapinfo xterm* ti=\E[?1049h:te=\E[?1049l 的作用

    这是一个非常经典的 `termcapinfo` 配置，通常出现在 `screen` 或 `tmux` 的配置文件（如 `.screenrc`）中。

    它的核心作用是**修复终端（特别是 xterm 系列）在退出全屏程序（如 Vim、Less、Top）后，屏幕残留内容或滚动混乱的问题**。

    为了让你彻底理解，我们分三部分拆解：

    ### 1. 基本语法结构
    `termcapinfo xterm* ti=... : te=...`

    - **`termcapinfo`**：告诉终端模拟器（这里是 screen/tmux）如何将内部的转义序列转换为外部真实终端的转义序列。
    - **`xterm*`**：通配符，匹配所有以 `xterm` 开头的终端类型（如 `xterm-256color`、`xterm-kitty` 等）。
    - **`ti=...`**：定义进入“全屏应用程序”模式时发送的指令（`ti` 即 `terminal initialize` 或 `smcup`）。
    - **`te=...`**：定义退出“全屏应用程序”模式时发送的指令（`te` 即 `terminal end` 或 `rmcup`）。

    ---

    ### 2. 转义序列的具体含义

    - **`\E[?1049h`**：
      - `\E` 代表 ESC 键（ASCII 27），即转义字符。
      - `[?1049h` 是 DEC 私有模式（Private Mode）的开启指令。
      - **作用**：**保存当前屏幕内容**（存入备用缓冲区）并**切换到备用屏幕**（Alternate Screen）。全屏程序（如 Vim）会在这个干净的备用缓冲区里绘制界面。

    - **`\E[?1049l`**：
      - `l` 代表关闭该模式（low）。
      - **作用**：**切回主屏幕**（Normal Screen）并**恢复之前保存的屏幕内容**。这样当你退出 Vim 时，你会看到打开 Vim 之前的 `bash` 提示符和输出，而不是 Vim 残留的代码界面。

    ---

    ### 3. 为什么要显式写这个配置？（解决历史遗留问题）

    在早期的 Xterm 实现中，`smcup`（进入全屏）和 `rmcup`（退出全屏）默认使用的是 **`\E[?47h` 和 `\E[?47l`**（老式备用屏幕）。

    **`1049` 是 `47` 的增强版**。区别在于：

    - **`47`**：只切换屏幕，**不保存**光标位置和属性。
    - **`1049`**：切换屏幕的同时，**保存并恢复**光标位置、字符集和视频属性。

    **为什么需要强制覆盖？**
    有些系统的 terminfo 数据库（`xterm` 条目）可能因为年代久远或自定义，仍然默认使用 `47` 而不是 `1049`。

    **加上这行配置后，强制效果如下：**

    - 当你进入 Vim 时：屏幕完全清空并保存当前 Bash 界面。
    - 当你退出 Vim 时：Bash 界面**原封不动**地恢复，且光标回到你敲 `vim` 命令之前的那一行，终端滚动历史（Scrollback）也不会被 Vim 的内容污染。

    ---

    ### 4. 补充场景（如果你不用 Screen/Tmux）

    - **如果你在原生终端（不带 screen）下**：这行配置无效，因为 `termcapinfo` 是 screen 特有的指令。原生终端直接读取系统的 `terminfo` 数据库，通常现代终端（GNOME Terminal, iTerm2, Windows Terminal）默认已经支持 `1049`，不需要配置。
    - **如果你在 Tmux 下**：Tmux 的等效配置通常是 `set -g terminal-overrides "xterm*:smcup@:rmcup@"`（强制禁用备用屏幕），因为 Tmux 自己管理滚动缓冲区，不希望内部程序切换屏幕导致混乱。

    ---

    ### 总结一句话
    **这行配置强制让 Screen 告诉 Xterm：进出全屏程序时，必须使用“保存/恢复主屏幕内容”的现代标准（1049），以防止退出 Vim 后屏幕残留垃圾或丢失之前的命令行历史。**

* 详细讲讲 screen 中 altscreen on 的作用

    `altscreen on` 这个设置，是用来解决一个在 `screen` 中很常见但又有点恼人的小问题：当你退出 `vim`、`less` 或 `top` 这类全屏程序后，它们的界面内容仍然会“粘”在终端上，挡住了你之前看到的内容。

    ### 问题的根源：备用屏幕 (Alternate Screen)

    要理解这个设置，首先要了解终端的“备用屏幕”机制。

    *   **是什么**：终端（如 Terminal.app, iTerm2, PuTTY）通常维护着两个独立的显示缓冲区：**主屏幕 (Main Screen)** 和**备用屏幕 (Alternate Screen)**。
    *   **怎么用**：像 `vim`、`less`、`htop` 这样的全屏交互式程序，在启动时会切换到备用屏幕，在这个全新的“画布”上绘制界面。程序退出时，会切换回主屏幕，让一切恢复原样，仿佛程序从未运行过。这样你就能看到启动程序前执行的命令和输出，体验非常流畅。

    ### `screen` 的默认行为：为什么界面会“残留”

    问题就出在 GNU Screen 本身也是一个终端模拟器。在默认情况下（`altscreen off`），Screen 会告诉其中的程序：“我不支持备用屏幕功能”。

    因此，当 `vim` 等程序试图切换屏幕时，这个请求实际上被忽略了。程序只是在当前屏幕（Screen 提供的主视图）上直接绘制和擦除内容。如果程序退出时没有完美地清理自己画的“画布”，它的界面就会残留下来，覆盖了你之前看到的 shell 内容。

    ### `altscreen on`：解决之道

    `altscreen on` 这个指令，就是明确地告诉 Screen：“请启用备用屏幕支持”。

    这样一来，当你在 Screen 中运行 `vim` 时，行为就和你直接在终端里运行一样了：
    1.  `vim` 启动，Screen 为其切换到内部的备用屏幕。
    2.  你编辑文件，`vim` 占满整个窗口。
    3.  你退出 `vim`，Screen 自动切回主屏幕。
    4.  **结果**：`vim` 的界面消失了，你启动 `vim` 之前的 shell 内容完好无损地回来了。

    ### 如何启用

    要让这个设置永久生效，只需将下面这行配置添加到你的 `~/.screenrc` 文件中即可：

    ```text
    altscreen on
    ```

    如果你只想在当前会话中临时测试一下，可以在 Screen 会话中按下 `Ctrl-a`，然后输入 `:altscreen on` 并回车。

* nonblock on 那这介呢

    `nonblock on` 这个命令的作用是**防止 `screen` 因为输出目标（比如你的 SSH 客户端）突然变慢或失去响应，而导致 `screen` 里运行的程序被卡死**。

    ### 为什么需要这个命令？

    `screen` 内部运行的程序产生输出时，需要将内容发送到你的终端显示。默认情况下（`nonblock off`），如果 `screen` 发现终端暂时无法接收数据，它会**一直等待**直到终端恢复，这个过程中，整个 `screen` 窗口和其中运行的程序都会被阻塞。

    这种情况通常发生在：
    *   你按下了 `Ctrl+S` 锁住了终端。
    *   网络连接突然变慢或中断，但 SSH 会话还没完全断开。
    *   你的 SSH 客户端（如 PuTTY）暂时失去响应。

    ### `nonblock on` 做了什么？

    开启 `nonblock on` 后，`screen` 的行为会改变：
    1.  **设定超时**：当 `screen` 发现终端无法接收数据时，它不会无限期等待。`on` 等同于设置了一个 **1秒** 的超时时间。
    2.  **超时后放弃**：如果 1 秒后终端依然无法接收数据，`screen` 会将这个终端标记为“阻塞”状态，并**停止向它发送输出数据**。
    3.  **程序继续运行**：最关键的是，`screen` 内部运行的程序**不会被阻塞**，可以继续在后台运行。
    4.  **恢复与刷新**：一旦终端恢复接收数据，`screen` 会解除阻塞状态，并自动刷新显示最新的窗口内容，让你看到之前的输出。

    ### 典型使用场景

    这个设置对于需要持续运行、不能因网络波动而中断的程序特别重要，例如：

    *   **IRC 客户端 (如 irssi)**：如果 `screen` 因为网络问题被阻塞，irssi 就无法及时响应服务器的 Ping 请求，导致被服务器断开连接。开启 `nonblock on` 可以避免这种情况。
    *   **远程下载或长时间任务**：确保在 SSH 连接不稳定时，`screen` 中的下载或计算任务不会因为输出阻塞而挂起。

    ### 如何设置？

    你可以在 `screen` 会话中临时启用（对当前窗口生效）：
    ```text
    Ctrl-a :nonblock on
    ```

    也可以将其写入 `~/.screenrc` 配置文件，让所有新窗口都默认启用：
    ```text
    nonblock on
    ```

    > **注意**：`nonblock on` 与 `defnonblock on` 功能相同，`defnonblock` 主要用于配置文件中，作为新窗口的默认设置。另外，这个功能也有一个权衡：当终端恢复时，可能会丢失少量无法被缓存的内容，但对于大多数追求程序稳定性的场景，这个代价是值得的。

* 讲讲 infocmp screen.xterm-256color > screen.terminfo 的作用

	这个命令 `infocmp screen.xterm-256color > screen.terminfo` 的作用是**查询名为 "screen.xterm-256color" 的终端类型描述，并将其以人类可读的文本格式重定向并保存到 `screen.terminfo` 文件中**。

	这个命令通常是 **`tic` 命令的“前置准备步骤”**，目的是为了导出一个终端描述文件进行修改或备份。让我们也像拆解上个命令一样，把它剖析开来看：

	### 🧐 命令拆解

	*   **`infocmp`**：这是 **terminfo 信息比较/查询工具**（terminfo comparison tool）的命令。它的核心作用是从系统的 terminfo 数据库中读取已编译好的终端描述（二进制格式），并将其以人类可读的源格式（Source Format）输出到终端上。它还能用来比较两个终端类型之间的差异。

	*   **`screen.xterm-256color`**：这是要查询的**终端类型名称**。这是一个非常常见的复合终端类型，它通常表示一个运行在支持 256 色（xterm-256color）的 xterm 终端模拟器内部的 GNU Screen 会话。它结合了 `screen` 的基础能力和 `xterm-256color` 的颜色特性。`infocmp` 会在 terminfo 数据库的众多条目中精确定位这个名称。

	*   **`>`**：这是 Shell 的**输出重定向符**。它将 `infocmp` 原本要打印到屏幕（标准输出）的内容，完整地写入到 `screen.terminfo` 这个文件中。如果文件已存在，它会被覆盖。

	*   **`screen.terminfo`**：这是**输出文件的名称**。这是一个约定俗成的文件名，表示这是一个关于 "screen" 的 terminfo 源文件。你可以把它理解为 **`tic` 命令的“原材料”**。

	### 💡 命令的实际效果

	当你执行这个命令时，会发生以下两件事：

	1.  **提取数据**：`infocmp` 会在系统的 terminfo 数据库中（通常是 `/etc/terminfo` 或 `/usr/share/terminfo`）找到名为 `screen.xterm-256color` 的条目。
	2.  **格式转换与保存**：它将该条目的二进制描述转换回文本格式（即 `tic` 命令能读懂的那类源文件），并通过重定向将所有文本内容保存到当前目录下一个叫做 `screen.terminfo` 的文件里。

	### 🔗 它和 `tic` 命令的关系

	如果你把上次讲的 `tic -x /usr/share/terminfo/s/screen` 和这个命令连起来看，就能明白一个完整的操作流程：

	1.  **导出**：先用 `infocmp screen.xterm-256color > screen.terminfo` 将系统中已有的终端描述**导出**为文本文件。
	2.  **修改**：你可以用文本编辑器（如 `vim`）打开这个 `screen.terminfo` 文件，根据需求添加、删除或修改其中的能力（比如调整功能键映射、更改颜色支持等）。
	3.  **编译安装**：再用 `tic -x screen.terminfo` 将修改后的文本文件**编译**并安装回 terminfo 数据库，从而覆盖或创建新的终端类型描述。

	### 💡 常见使用场景

	*   **终端定制**：许多用户会用这个方法导出一个终端描述，然后修改其中的转义序列，让功能键（如 F1-F12）或小键盘在 `screen` 下表现得更符合预期。
	*   **诊断与检查**：直接查看 `screen.xterm-256color` 具体支持哪些能力（比如 `colors` 字段是否为 256），省得去翻阅晦涩的 terminfo 手册。
	*   **备份**：在对系统终端数据库进行修改前，先导出一份原始描述的备份。

	### ⚠️ 需要注意

	*   **文件内容**：生成的 `screen.terminfo` 文件通常以 `screen.xterm-256color|...` 开头，其中 `|` 后面的部分是它的别名（alias）。编辑时要注意语法的正确性。
	*   **备用名称**：如果你的系统没有 `screen.xterm-256color` 这个条目，`infocmp` 会报错。此时你可以尝试查询更基础的名称，比如 `screen` 或 `screen-256color`。

* 讲讲 tic -x screen.terminfo 这个命令的作用

	这个命令 tic -x screen.terminfo 的作用是将当前目录下名为 screen.terminfo 的终端描述源文件编译成二进制格式，并安装到系统的 terminfo 数据库中，同时允许识别非标准的自定义能力。

	建议：

	备份原始描述：在覆盖系统数据库之前，先用 infocmp screen.xterm-256color > backup.terminfo 做备份。

* 讲讲 tic -x /usr/share/terminfo/s/screen 的作用

	这个命令 `tic -x /usr/share/terminfo/s/screen` 的作用是**编译一个名为 "screen" 的终端描述文件，并将其安装到系统的 terminfo 数据库中，同时允许识别非标准的自定义能力。**

	为了理解整个过程，我们可以把这个命令拆解成几个部分来看：

	### 🧐 命令拆解

	*   **`tic`**：这是 **terminfo 编译器**（terminfo compiler）的命令。它的核心作用是把人类可读的终端描述文本（Source Format）编译成程序能高效读取的二进制格式（Compiled Format），并安装到 terminfo 数据库中。像 `vim`、`screen`、`tmux` 这类依赖 `ncurses` 库的程序，正是通过这个数据库来了解特定终端类型支持哪些能力（如移动光标、改变颜色等）。

	*   **`-x`**：这是一个关键选项，意思是 **"允许非标准能力"**（Treat unknown capabilities as user-defined）。`tic` 通常只识别标准 terminfo 定义的能力。如果加上 `-x`，当它遇到不认识的扩展能力时，不会报错，而是会将其作为用户自定义能力处理并纳入编译。这能确保一些增强型终端描述或特殊配置不会因为包含非标准字段而编译失败。

	*   **`/usr/share/terminfo/s/screen`**：这是要编译的**源文件路径**。它通常是一个文本文件，其中包含了 "screen" 这个终端类型的所有能力描述。在 terminfo 数据库的目录树结构中，文件一般按终端名称的首字母存放在子目录下，所以 "screen" 的描述文件就在 `s/` 子目录中。

	### 💡 命令的实际效果

	当你执行这个命令时，`tic` 程序会：
	1.  读取 `/usr/share/terminfo/s/screen` 这个源文件。
	2.  根据 `-x` 选项，宽容地处理其中的所有能力定义，包括非标准的。
	3.  将处理后的描述编译成二进制格式。
	4.  默认情况下，将编译好的文件安装到系统的 terminfo 数据库中（通常是 `/etc/terminfo` 或 `/usr/share/terminfo` 等位置，具体取决于系统配置和权限）。

	这个操作的一个常见场景是，当系统自带的 "screen" 终端描述缺失、不完整或者需要更新时，通过手动编译来确保它可用。

	### ⚠️ 需要注意

	*   **权限**：写入系统级别的 terminfo 数据库（如 `/usr/share/terminfo`）通常需要 root 权限。如果权限不足，`tic` 可能会尝试将文件写入用户目录下的 `$HOME/.terminfo`。
	*   **验证**：如果你想先检查文件是否有语法错误，而不实际安装，可以使用 `-c` 选项（例如 `tic -c -x /usr/share/terminfo/s/screen`）。

* screen 是否有记录它接收到的重绘指令序列的 log？

	GNU Screen 确实提供了一个非常强大的底层调试功能，能够把所有接收到的数据和控制指令序列（包括 Vim 发送给它的所有重绘、移动光标等转义序列）全部记录到日志文件中。

	这个功能叫做 **输入/输出日志（Input/Output Logging）**，在 Screen 中被称为 `log` 和 `log_raw`。

	如果你想揪出 Vim 到底给 Screen 发了什么，以及 Screen 为什么没反应，可以用以下方法抓取 Log：

	### 1. 如何开启重绘指令日志

	因为我们要抓取的是**未经过滤的原生控制序列（Raw Control Sequences）**，所以必须同时开启标准日志和 Raw 模式：

	1. 正常打开 `screen`。
	2. 按快捷键 **`Ctrl + A`**，然后输入 **`:`**（冒号），进入 Screen 的命令行模式。
	3. 输入以下命令并回车，开启 Raw 模式（记录原始转义字符）：
	```text
	log_raw on

	```


	4. 再次按 **`Ctrl + A`**，然后按下 **`H`**（大写的 H，即 `Shift + H`）。
	* 此时屏幕底部会提示：`Creating logfile "screenlog.0"`。



	现在，Screen 已经开始把你终端上发生的所有一举一动（包括不可见的重绘控制符）记录到当前目录下的 `screenlog.0` 文件中了。

	---

	### 2. 复现并抓取错误

	1. 在当前 Screen 窗口中打开 `vim`。
	2. 疯狂做一些会导致页面混乱的操作（比如滚屏、快速切换模式）。
	3. 当屏幕**刚一变混乱**的时候，立刻按下 **`Ctrl + A`**，然后按 **`H`** 关闭日志录制（保护现场，防止后面的重绘日志把错误的日志冲掉）。

	---

	### 3. 如何分析 `screenlog.0`

	由于控制序列包含大量的不可见字符（如 `ESC` 键 `\033` 或 `^[`），直接用 `cat` 或普通的文本编辑器是看不出端倪的。你需要使用 `od`、`hexdump` 或者 `less` 来查看。

	**使用 `less` 查看（推荐）：**

	```bash
	less -r screenlog.0

	```

	在日志中，你会看到大量类似这样的底层指令：

	* `^[[H` （将光标移动到左上角）
	* `^[[2J` （清空整个屏幕）
	* `^[[m` （重置颜色）

	#### 🔍 核心排查方向：

	你可以重点搜索 Vim 发送的清屏重绘序列。如果页面混乱时，日志里**只有**文字内容，**完全没有**出现 `^[[2J` 或 `^[[H` 这样的清屏/移动光标序列，那就坐实了：**Vim 压根就没有向 Screen 发出重绘指令**（说明是 Vim 内部的 `term` 驱动判断失误）。

	相反，如果日志里**有**这些清屏指令，但外层的 Cygwin 却没有清屏，那就说明：**Screen 接收到了，但在转发给 Cygwin 时把这个信号弄丢了**（Screen 的 termcap 问题）。

* tmux 相比 screen，优势在哪里

    `tmux` 和 `screen` 都是终端复用器（terminal multiplexer），都能实现会话保持、窗口管理、断线重连等功能。

    不过，现在绝大多数 Linux 用户都会推荐 **tmux**，因为它在架构、功能和可扩展性方面都比 `screen` 更现代。

    下面从几个方面比较。

    | 功能       | tmux       | screen  |
    | -------- | ---------- | ------- |
    | 开发活跃度    | ⭐⭐⭐⭐⭐ 持续维护 | ⭐⭐ 更新较少 |
    | 配置能力     | 很强（类似 vim） | 一般      |
    | 窗口布局     | 非常灵活       | 比较有限    |
    | Pane（分屏） | 原生支持       | 支持较弱    |
    | 脚本控制     | 很强         | 一般      |
    | 鼠标支持     | 完整         | 较弱      |
    | 状态栏      | 可高度自定义     | 简单      |
    | 插件生态     | 丰富         | 基本没有    |

    ---

    ## 1. Pane（分屏）能力更强

    这是 tmux 最大的优势之一。

    例如：

    ```
    +-----------------------+
    |           |           |
    |  vim      |  htop     |
    |           |           |
    +-----------+-----------+
    |      shell            |
    +-----------------------+
    ```

    tmux 可以方便地：

    ```
    Ctrl+b %
    ```

    左右分屏

    ```
    Ctrl+b "
    ```

    上下分屏

    而 screen 虽然也支持 split，但：

    * 操作复杂
    * 切换不方便
    * 功能没有 tmux 丰富

    ---

    ## 2. Layout 自动调整

    tmux 可以自动重新排列 Pane。

    例如：

    ```
    Ctrl+b Space
    ```

    循环不同布局：

    ```
    even-horizontal

    even-vertical

    main-horizontal

    main-vertical

    tiled
    ```

    screen 基本没有这些能力。

    ---

    ## 3. 配置更现代

    tmux：

    ```
    ~/.tmux.conf
    ```

    例如：

    ```bash
    set -g mouse on

    setw -g mode-keys vi

    bind r source-file ~/.tmux.conf
    ```

    修改后：

    ```
    Ctrl+b r
    ```

    立即重新加载。

    screen：

    ```
    ~/.screenrc
    ```

    功能相对少很多。

    ---

    ## 4. 鼠标支持

    tmux：

    ```
    set -g mouse on
    ```

    以后可以：

    * 点击 Pane
    * 拖动滚动
    * Resize Pane
    * 复制

    screen 对鼠标支持一直不好。

    ---

    ## 5. Copy Mode 更舒服

    tmux 可以使用 vi 风格：

    ```bash
    setw -g mode-keys vi
    ```

    之后：

    ```
    Ctrl+b [

    ```

    进入复制模式。

    操作和 vim 一样：

    ```
    hjkl
    w
    b
    gg
    G
    /
    ?
    ```

    复制：

    ```
    Space
    Enter
    ```

    screen 默认按 Emacs 风格，更不符合很多开发者习惯。

    ---

    ## 6. Session 管理更方便

    例如：

    ```
    tmux ls
    ```

    ```
    work
    test
    gpu
    debug
    ```

    进入：

    ```
    tmux attach -t gpu
    ```

    新建：

    ```
    tmux new -s gpu
    ```

    重命名：

    ```
    Ctrl+b $
    ```

    删除：

    ```
    tmux kill-session -t gpu
    ```

    整体体验比 screen 更自然。

    ---

    ## 7. 命令接口更强

    tmux 几乎所有操作都有命令。

    例如：

    创建窗口：

    ```bash
    tmux new-window
    ```

    发送命令：

    ```bash
    tmux send-keys \
        -t work:0 \
        "python train.py" C-m
    ```

    切 Pane：

    ```bash
    tmux select-pane -L
    ```

    Resize：

    ```bash
    tmux resize-pane -L 10
    ```

    所以很多自动化工具都会控制 tmux。

    screen 虽然也支持 `screen -X`，但命令远没有 tmux 丰富。

    ---

    ## 8. 更好的脚本能力

    例如：

    ```bash
    tmux new-session -d -s train

    tmux split-window -h

    tmux send-keys "python train.py" C-m

    tmux send-keys -t 1 "watch nvidia-smi" C-m
    ```

    一条脚本直接生成：

    ```
    +--------------------+
    | python train.py    |
    |                    |
    +---------+----------+
    |watch    | shell    |
    +----------+---------+
    ```

    很多 CI、远程开发、机器学习训练都会这么做。

    ---

    ## 9. 插件生态

    tmux 有成熟的插件管理器：

    **Tmux Plugin Manager（TPM）**

    常见插件：

    * tmux-resurrect（保存/恢复 Session）
    * tmux-continuum（自动恢复）
    * tmux-yank（复制到系统剪贴板）
    * tmux-prefix-highlight
    * catppuccin 主题

    screen 几乎没有成熟插件生态。

    tmux 可以显示：

    ```
    CPU

    GPU

    Battery

    Load
    Time
    ```

    例如：

    ```
    [GPU 73%][RAM 12G][CPU 35%][12:35]
    ```

    screen 的状态栏功能比较基础。

    ## 11. 性能

    两者都很轻量。

    一般来说：

    * 内存占用都只有几 MB。
    * CPU 占用都很低。

    对于 SSH、日志查看、开发、训练等场景，性能差异通常可以忽略。

    ## 什么时候还会用 screen？

    虽然 `screen` 已经不是多数人的首选，但仍有一些场景适合：

    * **老旧 Linux 系统**：很多企业发行版默认安装 `screen`，而没有 `tmux`。
    * **极简需求**：如果只需要启动一个长期运行的程序、断线后重新连接，`screen` 已经足够。
    * **兼容历史脚本**：一些老环境和运维脚本仍然依赖 `screen`。

    ## 总结

    如果今天开始学习终端复用器，我建议直接学习 **tmux**。它相较于 `screen` 的主要优势是：

    * 更强大的分屏（Pane）和布局管理。
    * 更现代、可读性更好的配置文件。
    * 更完善的鼠标支持和 `vi` 风格复制模式。
    * 丰富的命令接口，便于自动化脚本控制。
    * 活跃的插件生态和高度可定制的状态栏。
    * 社区活跃，文档和教程更加丰富。

    对于软件开发、远程服务器管理、机器学习训练、多任务调试等日常工作，`tmux` 通常是更高效、更值得投入学习成本的选择。
    
* screen 如何切换窗口顺序

    在 GNU `screen` 中，默认情况下它会按照你创建窗口的顺序自动分配编号（0, 1, 2...）。如果你想切换、交换或重新排列这些窗口的顺序，可以使用以下几种方法：

    1. 修改当前窗口的编号（最常用）

        如果你想直接改变某个窗口的编号（例如把当前的 3 号窗口改成 1 号），可以使用 `number` 命令：

        1. 切换到你想修改编号的窗口。
        2. 按下快捷键 `Ctrl + A`，然后输入冒号 `:` 进入命令模式。
        3. 输入 `number <新编号>` 并回车。
        * *例如：* `:number 1`

        > ⚠️ **注意：** 如果新编号已经被其他窗口占用了，`screen` 会提示失败。你需要先把占用该编号的窗口改到别的数字上。

    ---

    2. 交换两个窗口的编号

        如果你用的是较新版本的 `screen`，它支持直接交换（swap）两个窗口的顺序，而不需要担心编号被占用的问题：

        1. 切换到你想交换的窗口。
        2. 进入命令模式：`Ctrl + A` 接着输入 `:`。
        3. 输入 `number <目标编号>`。如果该编号已被占用，互换会自动发生。

        如果你的版本不支持自动互换，可以手动用一个“临时编号”中转：

        * 将窗口 A 改为 99 号：`:number 99`
        * 将窗口 B 改为 窗口 A 的原编号：`:number <A的原编号>`
        * 将窗口 A 改为 窗口 B 的原编号：`:number <B的原编号>`

    ---

    3. 在窗口列表中可视化切换

        如果你只是想**查看**并快速跳转，而不是永久改变它们的数字顺序：

        1. 按下 `Ctrl + A` 然后按 `"`（双引号）。
        2. 屏幕上会弹出一个当前所有窗口的列表。
        3. 使用 **方向键** 或 **j/k** 键上下移动光标，按 **回车键 (Enter)** 即可直接切换到该窗口。

    ---

    4. 快速左右轮流切换

        如果你只是想在相邻的窗口之间临时切着玩，用这两个快捷键最快：

        * `Ctrl + A` 然后按 `n` (Next)：切换到**下一个**编号的窗口。
        * `Ctrl + A` 然后按 `p` (Previous)：切换到**上一个**编号的窗口。

* 使用 screen 后，进入 ctrl + a, [ 的复制模式，然后使用两次空格复制完数据后，进入 vim 中，insert 模式，然后使用 ctrl + a, ] 粘贴。但是由于 vim 有 autoindent，此时会造成输出里有大量的自动缩进。如何解决这个问题？

    这是一个非常经典且让人头疼的“梯形缩进”问题。原因在于 Vim 的 `autoindent`（自动缩进）误把 `screen` 粘贴进来的文本当成了你正在**手动输入**的代码，结果每一行都在前一行的基础上叠加上一次缩进。

    解决这个问题的最佳方案是利用 Vim 的 **`paste` 模式**。在这个模式下，Vim 会暂时关闭所有的自动缩进和快捷键映射，完美还原复制的内容。

    以下是几种最实用的解决方法：

    **方法一：临时开启 paste 模式（最常用）**

    在准备粘贴之前，先在 Vim 中执行以下步骤：

    1. 按 `Esc` 退出当前的 Insert 模式，回到 Normal 模式。
    2. 输入 `:set paste` 并回车。
    3. 按 `i` 进入 Insert 模式（此时底部会显示 `-- INSERT (paste) --`）。
    4. 使用 `Ctrl + a, ]` 粘贴你的文本。
    5. 粘贴完成后，按 `Esc` 回到 Normal 模式，输入 `:set nopaste` 恢复普通的编辑模式。

    **方法二：绑定快捷键快速切换（一劳永逸）**

    频繁输入 `:set paste` 略显麻烦，你可以在你的 `~/.vimrc` 文件中加一行配置，绑定一个快捷键（比如 `F2`）来一键切换：

    ```vim
    set pastetoggle=<F2>
    ```

    **使用方法：**

    在 Vim 的 Insert 模式或 Normal 模式下，直接按下 `F2` 键，就能在“普通模式”和“粘贴模式”之间无缝切换，省去了手动敲命令的麻烦。

    screen 的复制缓冲区与文件的交互：

    * 方案一：让 screen 把复制内容自动写进文件，Vim 直接读取

        这是纯终端环境下的终极魔改。我们可以让 `screen` 在复制时，把内容同步写到一个临时文件里，然后让 Vim 去读这个文件。

        1. 在你的 `~/.screenrc` 中加入这行配置（允许写出缓冲区）：

            ```screen
            bufferfile /tmp/screen-exchange
            ```

        2. 在 `~/.vimrc` 中写一个快捷键（比如映射到 `F3`）：

            ```vim
            nnoremap <F3> :r /tmp/screen-exchange<CR>
            ```

        **使用流程：**

        * 在 `screen` 中用 `Ctrl + a, [` 选中并复制文本。
        * 复制完成后，按下 `Ctrl + a, >`（这会把刚刚复制的内容写入 `/tmp/screen-exchange` 文件）。
        * 打开 Vim，在 **Normal 模式**下直接按 `F3`。Vim 会直接把文件内容读进来，**完全不经过 Insert 模式，因此绝对不会触发任何自动缩进！**

    * 方案二：利用 `xclip` / `xsel` 连通系统剪贴板（需要 X11 支持）

        如果你的 `screen` 运行在本地 Linux，或者通过 `ssh -X` 连接到服务器，你可以让 `screen` 复制的内容直接进入系统的 `+` 寄存器。

        在 `~/.screenrc` 中加入：

        ```screen
        bindkey -m ' ' eval 'stuff \015' 'writebuf' 'exec sh -c "xclip -sel clip /tmp/screen-exchange"'
        ```

        *(注：这串命令的意思是，当你用空格结束 screen 复制时，它自动把内容扔给系统的剪贴板 `xclip`)*

        一旦配置成功，`screen` 复制的内容就真的进入系统剪贴板了。此时你在 Vim 中（需要 Vim 支持 `+clipboard`），直接在 Normal 模式下敲 **`"+p`** 就能秒贴，同样不需要进入 Insert 模式，完美避开缩进问题。

    **总结来说：** 如果不想折腾复杂的配置，遇到 `screen` 粘贴时，最省心的方式依然是前面提到的 **方法二（用 `F2` 切换 `pastetoggle`）**。

    * screen append

        默认情况下，如果你直接使用 `Ctrl + a, >`（执行的是 `writebuf` 命令），`screen` 会**直接覆盖**旧的 `screen-exchange` 文件，而不会追加。

        但是，`screen` 内部其实提供了一个**内置的追加机制**。它允许你在**复制文本的阶段**就把内容“追加”到 `screen` 的内部缓冲区里，然后再通过 `Ctrl + a, >` 一次性写出去。

        这里有两种实现追加的方法：

        方法一：利用 Screen 的内置“追加复制”模式（推荐，纯原生）

        当你进入 `screen` 的复制模式后，可以通过以下操作实现追加：

        1. 按 `Ctrl + a, [` 进入复制模式。
        2. 移动光标到你想复制的起点，按下**第一下空格**（设置起点）。
        3. 移动光标到终点。**注意：此时不要按空格，而是按下大写的 `A` 键。**
        4. 按下 `A` 后，当前选中的文本就会追加（Append）到你上一次复制的内容后面，并自动退出复制模式。
        5. 此时再按下 `Ctrl + a, >`，写入到 `screen-exchange` 文件里的就是多次累加后的完整内容了。

        方法二：在 Shell 层面通过重定向追加（适合需要多次独立写入）

        如果你希望每次按 `Ctrl + a, >` 时，它在文件末尾追加，而不是覆盖，原生 `writebuf` 命令没有直接提供 `append` 参数。但我们可以通过在 `~/.screenrc` 中自定义一个快捷键，利用 shell 的 `>>` 符号强行追加。

        在你的 `~/.screenrc` 中加入这样一行：

        ```screen
        # 将 Ctrl + a, M 绑定为：将缓冲区写到临时文件，然后利用 cat 追加到 screen-exchange
        bind M eval "writebuf /tmp/screen-tmp" "exec sh -c 'cat /tmp/screen-tmp >> /tmp/screen-exchange'"
        ```

        **使用流程：**

        1. 正常用 `Ctrl + a, [` 和双击空格复制第一段。
        2. 按下 `Ctrl + a, M`（大写 M），内容被追加到 `/tmp/screen-exchange`。
        3. 复制第二段，再次按下 `Ctrl + a, M`……
        4. 这样你就可以无限追加。最后在 Vim 中直接用 `:r /tmp/screen-exchange` 一把读入。
        *(注：用这个方法前，记得先去 `/tmp/screen-exchange` 把旧文件删掉或清空，以免混入以前的历史数据。)*

* screen 如何将默认 shell 设置成 bash?

    在 `screen` 中，它默认通常会调用你的系统默认 Shell（可以通过 `echo $SHELL` 查看）。如果你想强制让 `screen` 启动时使用 **bash**，可以通过以下两种方式实现：

    **方法一：修改配置文件（永久生效，推荐）**

    这是最标准的方法。通过修改当前用户的 `~/.screenrc` 文件，指定 `shell` 参数。

    1.  打开（或创建）配置文件：
        ```bash
        vim ~/.screenrc
        ```
    2.  在文件中添加以下一行：
        ```text
        shell /bin/bash
        ```
        *如果你希望启动的是 **Login Shell**（这样会加载 `~/.bash_profile` 等配置），可以写成：*
        ```text
        shell -/bin/bash
        ```
        *(注意前面的横杠 `-`)*

    ---

    **方法二：启动时临时指定**

    如果你不想修改配置文件，只是偶尔想在 `screen` 里用 bash，可以在启动命令后直接加参数：

    ```bash
    screen /bin/bash
    ```

    ---

    **方法三：修改系统的默认 Shell（全局影响）**

    如果你希望无论是在 SSH 登录还是在 `screen` 里都默认使用 bash，可以直接修改账户的默认设置：

    ```bash
    chsh -s /bin/bash
    ```
    > **注意**：执行此命令后通常需要重新登录终端才能生效。

    ---

    **验证是否生效**

    进入 `screen` 后，你可以运行以下命令来确认当前的 Shell：

    ```bash
    echo $0
    ```
    如果输出是 `-bash` 或 `/bin/bash`，说明设置成功了。

    ---

    **小贴士：**

    如果你发现进入 `screen` 后 bash 的**颜色消失了**或者 **alias 失效了**，通常是因为 `screen` 默认启动的是非登录 Shell（Non-login shell）。此时建议使用方法一中的 `shell -/bin/bash`（带横杠的形式），这样它就会像你刚登录系统时一样，完整加载所有的环境变量。

* 我如何判断一个 terminal 窗口是否开了 screen ?

    有几种方法可以判断当前终端是否运行在 `screen` 会话中：

    1. 检查环境变量（最常用）

        ```bash
        echo $STY
        ```

        - **有输出**（如 `1234.tty1`）：在 screen 中
        - **空输出**：不在 screen 中

        `STY` 是 screen 会话特有的环境变量，记录当前会话名称。

        ```bash
        # 一行判断并输出结果
        [ -n "$STY" ] && echo "在 screen 中" || echo "不在 screen 中"
        ```

    2. 检查 TERM 变量

        ```bash
        echo $TERM
        ```

        - **screen 中**：通常显示 `screen` 或 `screen-256color`
        - **普通终端**：显示 `xterm`、`xterm-256color`、`linux` 等

        但这个方法**不够准确**，因为可以手动修改 TERM 变量。

    3. 查看父进程

        ```bash
        ps -p $PPID -o comm=
        ```
        - 输出 `screen`：在 screen 中
        - 输出其他（如 `bash`、`zsh`、`sshd`）：不在 screen 中

        更详细的父进程信息：
        ```bash
        ps -f -p $PPID
        ```

    4. 检查进程树

        ```bash
        pstree -p $$ | grep screen
        ```
        或
        ```bash
        ps -ef | grep -v grep | grep "screen"
        ```

    5. 检查终端内的按键行为

        在 screen 中，默认的转义键是 `Ctrl+A`：
        - 按 `Ctrl+A` 后按 `?`：会显示 screen 帮助（如果在 screen 中）
        - 按 `Ctrl+A` 后按 `d`：会脱离（detach）screen 会话

        **注意**：这个方法最直观，但会触发 screen 命令，可能产生副作用。

    **推荐判断方法**

    **最可靠**的方法是结合环境变量和父进程：

    ```bash
    if [ -n "$STY" ]; then
        echo "在 screen 会话中: $STY"
    elif ps -p $PPID -o comm= | grep -q screen; then
        echo "父进程是 screen"
    else
        echo "不在 screen 中"
    fi
    ```

    或者创建一个别名方便使用：

    ```bash
    alias isscreen='[ -n "$STY" ] && echo "In screen: $STY" || echo "Not in screen"'
    ```

    **类似工具的环境变量**

    其他终端复用工具也有类似的环境变量：
    - **tmux**：`echo $TMUX`（有值表示在 tmux 中）
    - **screen**：`echo $STY`（有值表示在 screen 中）

* 简述 screen 的用法

    `screen` 是一个 **终端复用器**。它最核心的价值在于：即使你断开了 SSH 连接，原本在服务器上运行的任务也不会中断，你可以随时“重连”回去查看进度。

    安装：`sudo apt install screen`

    以下是 `screen` 的常用操作指南：

    1. 核心操作流程

        通常的使用逻辑分为三步：**创建 -> 挂起（离开）-> 恢复**。

        | 操作步骤 | 命令 / 快捷键 | 说明 |
        | --- | --- | --- |
        | **新建窗口** | `screen -S <名字>` | 创建并进入一个自定义命名的会话 |
        | **离会话** | `Ctrl + a`, 然后按 `d` | **保持任务运行**，并退回到主终端 (Detach) |
        | **查看列表** | `screen -ls` | 查看当前后台运行的所有会话及其 ID |
        | **恢复会话** | `screen -r <名字或ID>` | 重新进入之前挂起的会话 (Resume) |
        | **彻底关闭** | `exit` 或 `Ctrl + d` | 在会话内输入，任务结束并销毁该会话 |

    2. 常用快捷键（在会话内部使用）

        所有 `screen` 指令都以 `Ctrl + a` (通常简写为 `C-a`) 作为前缀。

        * `Ctrl + a` + `d`：脱离当前会话（最常用）。

        * `Ctrl + a` + `k`：强制杀掉当前窗口。

        * `Ctrl + a` + `[`：进入**复制模式**。你可以用方向键翻页查看历史滚屏记录，按 `Esc` 退出。

        * `Ctrl + a` + `c`：在当前会话中创建一个新的子窗口。

        * `Ctrl + a` + `n / p`：在多个子窗口间切换（Next / Previous）。

    3. 进阶技巧

        * **清理无效会话**：如果程序崩溃导致会话显示为 `Dead`，使用 `screen -wipe` 清除。

        * **强制接管**：如果会话显示 `Attached`（可能在另一台电脑连着），用 `screen -d -r <名字>` 强制将其在那头断开并在此处恢复。

        * **日志记录**：使用 `screen -L` 启动会话，可以将所有输出自动保存到日志文件中。

    常见场景示例

    如果你要运行一个耗时 10 小时的脚本：

    1. 输入 `screen -S train_model`。

    2. 在新窗口里执行 `./script.sh`。

    3. 按 `Ctrl + a`, `d` 潇洒下班，关掉笔记本。

    4. 第二天回公司，输入 `screen -r train_model` 查看结果。

* 如何查看当前会话中子窗口的信息

    在 `screen` 的单个会话中，如果你通过 `Ctrl + a` + `c` 创建了多个子窗口（Window），可以通过以下几种方式查看这些窗口的信息：

    1. 调出窗口列表（最直观）

        按下快捷键：

        > **`Ctrl + a`** 然后按 **`"`** (双引号键)

        你会看到一个交互式的列表，列出了当前会话中所有活跃的子窗口。

        * **操作**：使用方向键上下移动，按 **Enter** 键即可切换到选中的窗口。
        * **标识**：列表里带有 `*` 的是当前正在查看的窗口，带有 `-` 的是上一个切换过的窗口。

    2. 状态栏底部查看

        按下快捷键：

        > **`Ctrl + a`** 然后按 **`w`**

        屏幕的左下角（或状态栏）会弹出一行临时文字，显示类似 `0* bash  1- bash  2 bash` 的信息：

        * **数字**：窗口编号。
        * **星号 `***`：代表你当前所在的窗口。
        * **减号 `-**`：代表上一个窗口。
        * **名称**：默认通常是你的 Shell 名称（如 `bash` 或 `zsh`）。

    3. 查看当前窗口的详细状态

        按下快捷键：

        > **`Ctrl + a`** 然后按 **`i`** (info)

        这会在屏幕底部显示当前窗口的详细参数，包括：

        * 窗口的分辨率（宽x高）。
        * 当前使用的字符编码（如 `UTF-8`）。
        * 窗口编号和名称。

    💡 进阶小贴士：给窗口起个名

    当子窗口多了以后，全叫 `bash` 会很难分辨。你可以给当前所在的窗口重命名：

    1. 按下 **`Ctrl + a`** 然后按 **`A`** (大写 A，即 `Shift + a`)。
    2. 删掉旧名字，输入新名字（如 `Database` 或 `Log-Monitor`）并回车。
    3. 之后再用 `Ctrl + a` + `"` 查看时，就能一眼认出哪个窗口是做什么的了。

* screen 在子窗口间切换

    如果你创建了多个子窗口（Window 0, 1, 2...），可以使用以下快捷键：

    * **快速轮换**：

        * `Ctrl + a` 然后按 `n` (Next)：切换到下一个窗口。

        * `Ctrl + a` 然后按 `p` (Previous)：切换到上一个窗口。

    * **按编号跳跃**：

        * `Ctrl + a` 然后按 **数字键 (0-9)**：直接跳到对应的窗口。

    * **回看切换**：

        * `Ctrl + a` 然后按 `Ctrl + a`：在当前和上一次查看的两个窗口之间快速来回切换。

* screen 分栏显示（类似 Vim 的 Split）

    `screen` 确实支持分栏，但逻辑和 Vim 略有不同：它先切分“区域（Region）”，然后你需要把“窗口（Window）”填进去。

    分栏操作：

    * 水平分栏：`Ctrl + a` 然后按 `S` (大写 S)。

    * 垂直分栏：`Ctrl + a` 然后按 `|` (竖线键，通常是 `Shift + \`)。

    分栏后的导航：

    * 移动光标：
    
        `Ctrl + a` 然后按 `Tab`。光标会在不同的分栏区域间跳动。

    * 填入内容：

        刚分好的栏是**空白**的。你需要跳到那个空白区域，然后：

        * 按 `Ctrl + a` 然后按 `c`：在那里创建一个新窗口。

        * 或者按 `Ctrl + a` 然后按数字（如 `0`）：把已有的窗口调到这个区域显示。

    关闭分栏：

    * **关闭当前栏**：`Ctrl + a` 然后按 `X` (大写 X)。

    * **只留当前栏（关闭其他所有）**：`Ctrl + a` 然后按 `Q` (大写 Q)。

    **Screen vs Tmux：一个真诚的建议**

    虽然你在问 `screen`，但不得不提的是，如果你对**分栏**、**鼠标拖拽**、**持久化布局**有较高要求，**Tmux** 是目前更现代、更强大的选择。

    | 特性 | Screen | Tmux |
    | --- | --- | --- |
    | **垂直分栏** | 部分老版本不支持 | 原生完美支持 |
    | **分栏持久化** | 重连后布局常消失 | 配合插件可完美恢复布局 |
    | **状态栏** | 需复杂配置 | 默认自带，信息丰富 |
    | **学习曲线** | 略低 | 略高（但回报大） |

    **快速上手分栏练习：**

    你可以现在尝试：

    1. `Ctrl + a` -> `S`（屏幕一分为二）。
    2. `Ctrl + a` -> `Tab`（跳到下半部）。
    3. `Ctrl + a` -> `c`（下半部出现新命令行）。

* screen 如何上下滚动查看 terminal 输出

    在 `screen` 中，你不能直接用鼠标滚轮或平常的 `Shift + PageUp/PageDown` 来滚动屏幕，因为 `screen` 接管了终端的显示缓存。

    要查看历史输出，你需要进入 `screen` 的 **“复制模式”（Copy Mode）**。

    1. 核心操作步骤

        1. **进入模式**：按下 **`Ctrl + a`**，然后按 **`[`** (左方括号)。

            * 此时你会发现光标变成了一个可以移动的方块，屏幕右上角可能会出现类似 `Copy mode` 的提示。

        2. **上下滚动**：

            * **方向键**：上/下/左/右 逐行移动。
            * **翻页键**：`PageUp` 和 `PageDown`。
            * **类 Vim 快捷键**（推荐）：

                * `k` / `j`：上 / 下移一行。
                * `u` / `d`：向上 / 向下翻半屏 (Up / Down)。
                * `b` / `f`：向上 / 向下翻全屏 (Back / Forward)。
                * `g`：跳到文件最开头；`G`：跳到文件最末尾。

        3. **退出模式**：按 **`Esc`** 键（连按两次更保险）或 `Enter`。

            * 退出后，光标会回到最底部，恢复正常的命令行输入状态。

    2. 为什么我滚不上去？（缓存设置）

        如果进入模式后发现只能向上翻几行就到头了，说明 `screen` 默认的**回滚缓冲区（Scrollback Buffer）**太小了。

        **临时解决方法**：

        启动 screen 时指定缓冲区行数（例如 10000 行）：

        ```bash
        screen -h 10000
        ```

        **永久解决方法**：

        修改你的 `~/.screenrc` 文件（如果没有就创建一个）：

        ```bash
        nano ~/.screenrc
        ```

        在文件中添加这一行：

        ```bash
        defscrollback 10000
        ```

        保存退出。以后新建的每一个 screen 窗口都会拥有 10000 行的历史记录。

    3. 进阶技巧：搜索关键词

        进入 `Copy mode` (`Ctrl + a` + `[`) 后，你可以像在 `less` 或 `vim` 中一样搜索内容：

        * 按 **`/`**：向下搜索字符串。
        * 按 **`?`**：向上搜索字符串。
        * 按 **`n`**：跳转到下一个匹配项。

    **提示**

    如果你使用的是 **Tmux**，操作逻辑非常相似：`Ctrl + b` + `[` 进入模式，然后可以用方向键或 Vim 键位滚动。

* 那么进入 copy 模式后，我该如何使得键盘像vim 那样复制内容？

    既然你已经进入了 **Copy Mode** (`Ctrl + a` + `[`），`screen` 本身的设计逻辑就非常接近经典的 `vi`。

    以下是在 `screen` 中像 Vim 一样“选取、复制、粘贴”的标准动作：

    1. 像 Vim 一样复制 (Copy)

        在 Copy Mode 状态下（光标可自由移动时）：

        1. **定位起始点**：使用 `h/j/k/l` 或方向键将光标移动到你想复制的文字开头。
        2. **标记开始 (Set Mark)**：按下 **`Space`** (空格键)。

            * 你会发现光标下的字符被反色显示，这表示选区开始了。


        3. **移动选区**：继续移动光标，你会看到被选中的部分变色。

        4. **完成复制 (Yank)**：再次按下 **`Space`** (空格键) 或 **`Enter`**。
            * 选中的内容现在已经存入 `screen` 的内部剪贴板（Exchange Buffer）中。

            * 此时会自动退出 Copy Mode，回到正常终端状态。

    2. 像 Vim 一样粘贴 (Paste)

        要把刚才复制的内容贴出来：

        * 按下快捷键：**`Ctrl + a`** 然后按 **`]`** (右方括号)。

            * 这就相当于 Vim 里的 `p`。

    3. 高级技巧：Vim 党必备

        如果你习惯了 Vim 的快捷键，在 Copy Mode 里这些也是通用的：

        * **`0` / `$**`：跳到行首 / 行尾。
        * **`w` / `b**`：按单词向后 / 向前跳。
        * **`g` / `G**`：跳到缓冲区最顶端 / 最底端。
        * **`/`**：向下搜索；**`?`**：向上搜索。

    ⚠️ 一个常见的“坑”：剪贴板隔离

    **请注意**：`screen` 的这个复制粘贴是**它内部的**。

    * 这意味着你用 `Space` 复制的内容，**无法直接**在你的 Windows 或 macOS 桌面通过 `Ctrl + V` 粘贴出来。

    * 它只能在同一个 `screen` 会话的其他窗口（Window）里通过 `Ctrl + a` + `]` 粘贴。

    > **如果你想把服务器上的内容复制到本地电脑：**

    > 建议按住键盘上的 **`Shift`** 键，然后直接用**鼠标拖拽选中**（这样会绕过 `screen` 的拦截，调用本地终端模拟器的复制功能），再在本地执行常规粘贴。
