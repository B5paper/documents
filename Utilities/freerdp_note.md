* xfreerdp 如何更改分辨率？

    xfreerdp 可以通过多种方式更改连接分辨率，以下是常用方法：

    ## 1. **使用 `/size` 参数**
    ```bash
    xfreerdp /v:服务器地址 /u:用户名 /size:1920x1080
    ```
    或指定百分比：
    ```bash
    xfreerdp /v:服务器地址 /u:用户名 /size:100%
    ```

    ## 2. **使用 `/w` 和 `/h` 参数**
    ```bash
    xfreerdp /v:服务器地址 /u:用户名 /w:1920 /h:1080
    ```

    ## 3. **使用动态分辨率（调整窗口时自动改变）**
    ```bash
    xfreerdp /v:服务器地址 /u:用户名 /dynamic-resolution
    ```

    ## 4. **常用分辨率示例**
    ```bash
    # 全屏
    xfreerdp /v:服务器地址 /u:用户名 /f

    # 4K分辨率
    xfreerdp /v:服务器地址 /u:用户名 /size:3840x2160

    # 2K分辨率
    xfreerdp /v:服务器地址 /u:用户名 /size:2560x1440

    # 高清
    xfreerdp /v:服务器地址 /u:用户名 /size:1280x720
    ```

    ## 5. **使用配置文件**
    在 `~/.freerdp/` 目录创建配置文件：
    ```bash
    xfreerdp /v:服务器地址 /u:用户名 /size:1920x1080 /sec:tls /clipboard /drive:home,/home/user
    ```

    ## 6. **完整示例命令**
    ```bash
    xfreerdp \
      /v:192.168.1.100 \
      /u:administrator \
      /p:密码 \
      /size:1920x1080 \
      /dynamic-resolution \
      /clipboard \
      +home-drive \
      /sound:sys:alsa \
      /network:auto
    ```

    ## 注意事项：
    - 某些服务器可能限制可用分辨率
    - `/dynamic-resolution` 需要服务器端支持
    - 全屏模式下可以使用 `Ctrl+Alt+Enter` 切换窗口模式
    - 使用 `+fonts` 参数可以获得更好的字体渲染

    查看所有选项：
    ```bash
    xfreerdp --help
    ```
* rdp 通信时的数据加密吗

    RDP (Remote Desktop Protocol) 在通信过程中是**加密的**。

    1. 默认加密 (Standard Encryption)

        * **传输层安全 (TLS/SSL):** 现代版本的 Windows Server 和 Windows Pro 客户端默认使用 **TLS 1.2** 或更高版本来加密 RDP 会话。
        * **私有 RDP 加密:** 在早期版本或某些配置下，RDP 也会使用微软私有的 **RDP Security Layer** 进行加密，但这种方式不如 TLS 安全和推荐。
        * **数据流保护:** 加密涵盖了整个 RDP 会话中的所有数据流，包括：
        * **屏幕图像和图形** (你看到的桌面画面)
        * **键盘和鼠标输入** (你的操作)
        * **剪贴板内容**
        * **文件传输** (如果你启用了驱动器映射)

    2. 网络级身份验证 (NLA)

        为了增强安全性，微软引入了 **网络级身份验证 (Network Level Authentication, NLA)**。

        * **功能:** NLA 要求用户在建立完整的 RDP 会话**之前**，就必须通过身份验证。
        * **好处:** 这可以防止许多拒绝服务 (DoS) 攻击和资源耗尽攻击，因为未经身份验证的连接无法占用服务器的资源来渲染完整的登录屏幕。当客户端连接时，NLA 使用 **Kerberos** 或 **TLS** 完成预认证。

    **总结**

    当您使用 RDP 连接时，您可以确信数据是经过加密的，可以抵御常见的窃听攻击。

    不过，为了最大化安全性，建议始终遵循以下实践：

    1. **使用 NLA:** 确保您的远程桌面主机启用了“只允许运行带网络级身份验证的远程桌面的客户端连接”选项。
    2. **复杂密码:** 使用复杂的密码，或考虑使用支持 MFA (多因素认证) 的 RDP 网关。
    3. **限制访问:** 与 SSH 类似，不要将 RDP 端口 ($3389$) 直接暴露给公网，而是通过 **VPN** 或 **RDP 网关** 进行连接。
    4. **更改默认端口:** 如果必须暴露，请像 SSH 一样更改 $3389$ 端口到非标准端口。

    您是否想了解如何配置 RDP 网关或如何在 Windows 中启用 NLA？

    * 那么登陆rdp时的user和passwd是加密的吗

        **是的，登录 RDP 时的用户名和密码是加密的。**

        在 RDP 连接中，用户名和密码属于**认证凭证（Authentication Credentials）**，它们是在 RDP 会话的认证阶段传输的。

        正如我们前面讨论的，RDP 会话在传输用户名和密码之前，会先完成以下步骤：

        1. **预认证/握手阶段：** 客户端和服务器协商加密算法。
        2. **建立加密通道：** 使用协商好的密钥和 TLS/SSL（或 RDP 私有加密）建立一个安全的、加密的通道。

        **用户名和密码的传输过程：**

        一旦加密通道建立起来，所有的后续通信，包括用户在登录界面输入的用户名和密码，都将在这个加密通道中传输。这意味着：

        * **数据是混淆的 (Obscured)：** 即使攻击者能够截获网络流量，他们看到的也只是无意义的乱码。
        * **窃听者无法获取凭证：** 攻击者无法从截获的数据包中直接提取出你的明文用户名和密码。

        **总结**

        用户名和密码在 RDP 登录过程中受到了加密协议的保护，因此是安全的。这与 SSH 登录的原理是一致的：所有的认证信息都是在建立了一个安全的加密隧道之后才发送的。

        **最主要的风险点在于：**

        如果 RDP 服务器或客户端本身存在漏洞，或者使用的加密协议版本过旧（例如只使用弱加密算法），那么加密的安全性才会受到威胁。这也是为什么建议始终保持操作系统和 RDP 客户端的更新，并启用像 **NLA**（网络级身份验证）这样的先进安全特性。NLA 甚至可以在登录屏幕出现前就完成大部分的身份验证，进一步保护了凭证。

    * 我看到ardp free软件中，有一个选项是“通过ssh的rdp”，这个是什么意思

        这是一个非常好的安全增强方式，通常被称为 **SSH 隧道 (SSH Tunneling)** 或 **SSH Port Forwarding（SSH 端口转发）**。

        `aRDP free` 软件（通常指的是在 Android 或其他平台上运行的 RDP 客户端）提供的“通过 SSH 的 RDP”选项，意味着它将 RDP 连接封装在一个**加密的 SSH 连接内部**。

        **SSH 隧道的工作原理**

        要理解这一点，你需要知道 RDP 和 SSH 通常是如何连接的：

        | **普通 RDP 连接** | **通过 SSH 隧道的 RDP 连接** |
        | --- | --- |
        | 1. 客户端直接连接到服务器的 RDP 端口（默认 $3389$）。 | 1. 客户端首先建立一条到 SSH 服务器的**加密连接**（默认 $22$ 或你更改的端口）。 |
        | 2. 整个 RDP 流量（包括加密后的屏幕数据）直接通过公网传输。 | 2. SSH 连接建立后，客户端请求 SSH 服务器将**本地端口**（例如 $3389$）的流量**转发**到**目标 RDP 服务器**的 $3389$ 端口。 |
        | 3. **RDP 加密**保护数据，但连接是直连的。 | 3. **RDP 流量被包裹在 SSH 隧道内部**。它现在经历了 **两层加密**。 |

        **为什么“通过 SSH 的 RDP”更安全？**

        虽然 RDP 本身已经加密了（如前所述），但通过 SSH 隧道进行封装提供了额外的和更强大的安全优势：

        1. 双重加密

            * **外层加密：** SSH 隧道提供的强大、经过验证的加密层。
            * **内层加密：** RDP 协议本身的加密层。
            * **效果：** 即使有人设法破解了其中一层加密，他们仍然需要破解另一层才能获取到数据。

        2. 隐藏 RDP 服务端口

            * **普通连接：** 攻击者可以直接扫描 $3389$ 端口，找到你的 RDP 服务。
            * **SSH 隧道：** 你可以将 RDP 端口**不对外开放**，只允许 **SSH 端口**（例如 $22$ 或自定义端口）暴露在公网上。RDP 流量只在 SSH 隧道内部流动，外部扫描无法直接发现 RDP 服务，**提高了隐蔽性**。

        3. 利用 SSH 的高级认证

            * 你可以强制要求 RDP 客户端使用**更安全的 SSH 密钥**进行身份验证来建立隧道，而不是只依赖 RDP 自身的密码。
            * 这样即使你的 RDP 密码被破解，攻击者仍然需要你的 SSH 密钥才能建立连接。

        **总结**

        “通过 SSH 的 RDP”是远程访问中的一个**最佳安全实践**。它的核心价值是：**将 RDP 流量隐藏在更安全、更灵活的 SSH 隧道中，实现更高的安全性和连接的隐蔽性。**

        如果您正在从不安全的公共网络访问您的服务器，强烈建议使用这种方式。
* 之前用 ubuntu 1 -> server -> ubuntu 2 -> win 11 rdp 3389 是正常的，可以用 rfx，但是现在在 win 11 上直接 ssh reverse tunnel 到 server，即 ubuntu 1 -> server -> win 11 rdp 却不能用 rfx 了

    这是一个非常典型的 RDP 协议“能力协商”失败的问题。在你的第一种链路（通过中间 Ubuntu 转发）中，RDP 流量表现为普通的点对点 TCP 流量；而在第二种链路（SSH 反向隧道）中，由于流量经过了本地回环接口（127.0.0.1）的转发，触发了 Windows 的安全或策略限制。

    以下是导致 RFX 无法启动的核心原因及其解决方案：

    1. 核心原因：Windows 对“回环地址”的策略限制

        当你在 Win 11 上使用反向隧道时，RDP 服务接收到的连接请求来自 `127.0.0.1`。Windows 默认认为本地回环连接不需要高性能图形加速（如 RemoteFX/高级编解码），或者出于安全考虑，在处理来自 localhost 的 RDP 请求时，某些高级功能会被降级或禁用。

    2. 解决方案：强制开启高性能策略

        你需要在 Win 11（受控端）中通过组策略明确告诉 Windows：**无论连接来自哪里，都必须使用高性能压缩和编解码。**

        步骤：

        1. 按下 `Win + R`，输入 `gpedit.msc` 并回车。
        2. 导航至：`计算机配置` -> `管理模板` -> `Windows 组件` -> `远程桌面服务` -> `远程桌面会话主机` -> `远程会话环境`。

    3. 必须配置项：

        * 启用 RemoteFX 编码：设置为 `已启用`。

        * 为远程桌面连接使用 WDDM 图形显示驱动程序：设置为 `已禁用`（有时 WDDM 驱动会干扰旧版 RFX，尝试切换此项）。

        * 配置 RemoteFX 数据的自适应图形编码：设置为 `已启用`，并选为 `优化服务器资源使用量`。

        * 为所有远程桌面会话使用硬件图形适配器：设置为 `已启用`。

    4. 执行 `gpupdate /force` 或重启电脑。

    注：

    1. 也可以使用改 ip 的方式，比如把 127.0.0.1 改成 192.168.100.1

        通过指定具体的局域网 IP（192.168.100.1），Windows 将其识别为真实的**网络路径**，从而完整激活了图形编解码加速、位图缓存和帧率优化功能。即使物理上流量还是流向自己，但逻辑上的“身份切换”绕过了回环接口的性能限制。

* freerdp 中`/`和`+`以及`-`的不同

    你提供的 `Usage` 和 `Syntax` 信息揭示了问题的关键：**你的 FreeRDP 版本对参数的解析逻辑与新版（v3）或通用文档不同。**

    仔细看这一行帮助信息：

    > `+toggle -toggle (enables or disables toggle, where '/' is a synonym of '+')`

    这句话的意思是：**`/` 等同于 `+`，只能用于“开启”功能。**

    因此，你输入 `/gfx:off` 实际上被解析器试图理解为“开启 gfx 并赋值为 off”，但这在旧版语法中是不合法的，导致了 Usage 报错；而默认情况下 `rdpgfx` 又是开启的（日志里显示 Loaded），最终导致了后续的段错误。

    核心解决方案

    要真正“禁用”引发崩溃的图形扩展，你必须使用 **`-` (减号)**。

    请尝试这条修正后的命令：

    ```bash
    xfreerdp /v:127.0.0.1 /port:6689 /u:wsdlh /p:xxxxxx -gfx -themes -wallpaper /cert-ignore
    ```

    为什么这样做有效？

    1. **`-gfx` (关键修改)**：你的版本中，`/gfx` 相当于 `+gfx` (强制开启)。只有 `-gfx` 才是强制禁用。日志显示程序在加载 `rdpgfx` 后崩溃，禁用它是解决段错误的核心。

    2. **`PDU_TYPE_DATA` 错误**：这通常是因为服务器发送了 FreeRDP 无法处理的高级图形指令（如 AVC444 或某些位图压缩）。`-gfx` 会迫使连接回退到更老、更稳定的传统 RDP 图形模式。

    3. **`/cert-ignore`**：针对 `Certificate verification failure` 警告，防止因证书交互导致连接挂起。
